# M01 后端维护说明

## 1. 技术栈

后端技术栈：

- Spring Boot `3.2.5`
- Java `21`
- Maven

当前模块 1 后端定位：

- 不是正式业务库
- 是“接口契约 + mock 数据服务 + 测试基线”

## 2. 响应结构

统一响应类型：

- `backend/src/main/java/cn/techstar/pvms/backend/common/ApiResponse.java`

统一格式：

```json
{
  "code": 0,
  "message": "success",
  "data": {}
}
```

目前没有统一异常拦截、鉴权和数据库接入。

## 4. 模块 1 接口清单

### 4.1 `GET /api/pvms/dashboard/stations-geo`

参数：

- `status?`
- `region?`
- `capacityRange?`

返回内容：

- 地图筛选项
- 状态汇总
- 电站列表

前端用途：

- GIS 点位绘制
- 地图筛选
- 重点关注电站
- 区域态势
- 调度摘要的部分聚合

### 4.2 `GET /api/pvms/dashboard/kpi-summary`

参数：

- `stationId?`

返回内容：

- 首页 8 个 KPI 指标

前端用途：

- 核心 KPI 看板

### 4.3 `GET /api/pvms/dashboard/power-curve`

参数：

- `stationId?`
- `date?`

返回内容：

- `actual`
- `plan`
- `forecast`
- `irradiance`
- `peakPowerKw`
- `deviationRate`

前端用途：

- 计划 vs 实际功率曲线

### 4.4 `GET /api/pvms/dashboard/station-ranking`

参数：

- `metric?`
  - `energy`
  - `hours`
  - `pr`

返回内容：

- 排名指标选项
- 单位
- 排名列表

前端用途：

- 发电排名

### 4.5 `GET /api/pvms/alarms/recent`

参数：

- `level?`
- `stationId?`

返回内容：

- 告警汇总
- 告警列表

前端用途：

- 实时告警列表
- 地图洞察区聚合

### 4.6 `GET /api/pvms/weather/current`

参数：

- `stationId?`

返回内容：

- 当前天气
- 三日预测

前端用途：

- 天气卡片

## 5. 数据来源

模块 1 后端 mock 数据全部集中在：

- `backend/src/main/java/cn/techstar/pvms/backend/module/dashboard/service/DashboardMockService.java`

当前内置的数据类型包括：

- `StationRecord`
- `AlarmRecord`
- `WeatherProfileRecord`

这个服务同时承担：

- mock 数据仓库
- 筛选逻辑
- 汇总计算
- 曲线构造
- 指标聚合

后续如果要上真数据，建议优先按下面顺序拆：

1. 把静态记录拆到 repository / fake repository
2. 把聚合计算留在 service
3. controller 只保留入参和出参封装

## 7. 当前维护注意事项

### 7.1 后端接口已完成，但前端开发态默认不消费这些接口

这是当前最容易误判的点。

如果后端接口返回改了，但前端页面没变，优先检查：

- `frontend/src/shared/host/bridge.js`

不是优先怀疑后端没生效。

### 7.2 模块 1 后端还没有做这些能力

当前没有：

- 数据库
- 持久化
- 统一异常处理
- 鉴权
- 配置化环境切换
- Swagger/OpenAPI

所以它现在是“可维护的契约服务”，不是正式业务后端。

## 8. 后续模块建议继续沿用的后端模式

后续模块建议继续保持：

1. 先定接口契约
2. 先写 controller test
3. 再写 service 实现
4. 每个模块都保留 `docs/modules/<module>.md`
5. 模块完成后，把最终维护说明写到 `doc/finished/`

这样后续模块能持续保持同样的维护粒度。
