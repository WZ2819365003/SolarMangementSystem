## 5. 当前 mock 规则

模块 2 后端 mock 不是随机拼数据，而是按聚合商语义建模：

- 同一资源单元共享天气剖面
- 同一资源单元共享总出力趋势
- 成员电站出力按权重和状态系数分摊
- `clusterRadiusKm` 用来表达资源单元空间聚合范围

状态系数当前定义：

- `normal -> 1.0`
- `warning -> 0.88`
- `fault -> 0.52`
- `offline -> 0.18`

## 6. 当前还没有做的事

当前后端还没有做：

- 真实数据库
- 真实天气采集
- 真实调度数据接入
- `date` 参数的正式查询逻辑
- OpenAPI/Swagger
- 统一异常与鉴权

所以它现在是“可维护的 mock 契约层”，还不是正式业务后端。

## 7. 测试要求

模块 2 后端开发必须保持测试先行。

当前测试文件：

- `backend/src/test/java/cn/techstar/pvms/backend/module/productionmonitor/controller/ProductionMonitorControllerTest.java`

当前覆盖：

- `meta`
- `overview`
- `output`
- `dispatch`
- `weather`

每新增一个接口或显著改动一个接口，必须先补测试，再改实现。

## 8. 后续演进建议

后续如果把 mock 切成真实服务，建议按这个顺序：

1. 保持 controller 路径和返回字段不变
2. 把静态资源单元数据拆到 repository 层
3. 保留资源单元聚合计算在 service 层
4. 单独引入天气采集适配层
5. 最后再接调度指令和执行回执
