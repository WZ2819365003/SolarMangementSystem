# 部署与运行说明

这份文档回答三个问题：

1. 本地怎么一键跑起来。
2. `docker compose` 怎么一键部署。
3. 这些脚本背后到底做了什么。

## 1. 运行前先认识仓库

仓库根目录最重要的目录有 4 个：

- `backend/`
  Spring Boot 3.2.5 后端。
- `frontend/`
  Vue 2 + Element UI 微前端子应用。
- `scripts/`
  本次补齐的一键运行、一键停止、一键推送当前分支和一键容器化脚本都在这里。
- `runtime-logs/`
  本地脚本启动后，前后端 stdout/stderr 和 pid 文件会写到这里。

## 2. 本地一键运行

### 2.1 脚本入口

新增的本地启动脚本有：

- `scripts/run-local.cmd`
- `scripts/run-local.ps1`
- `scripts/stop-local.ps1`
- `scripts/push-current-branch.cmd`
- `scripts/push-current-branch.ps1`

已有的单服务脚本有：

- `scripts/run-backend-dev.cmd`
- `scripts/run-frontend-dev.cmd`

### 2.2 推荐启动方式

在仓库根目录执行：

```powershell
.\scripts\run-local.cmd
```

如果你已经确认依赖都安装好了，也可以跳过安装检查：

```powershell
powershell -ExecutionPolicy Bypass -File .\scripts\run-local.ps1 -SkipInstall
```

### 2.3 这个脚本按顺序做了什么

`scripts/run-local.ps1` 的执行步骤如下：

1. 计算仓库根目录、前端目录、后端目录和日志目录。
2. 优先寻找 `C:\Users\<用户名>\tools\node-v14.16.0-win-x64\npm.cmd`。
3. 如果 `frontend/node_modules` 不存在，就先执行一次 `npm install --scripts-prepend-node-path=true`。
4. 通过 `cmd.exe /c scripts\run-backend-dev.cmd` 启动后端。
5. 通过 `cmd.exe /c scripts\run-frontend-dev.cmd` 启动前端。
6. 把前后端日志分别写到 `runtime-logs/local-dev/`。
7. 把 PID、URL、日志路径写到 `runtime-logs/local-dev/pids.json`，供停止脚本使用。

### 2.4 单服务脚本的职责

#### `scripts/run-backend-dev.cmd`

做的事情很简单：

1. 切到 `backend/`
2. 执行 `mvn.cmd spring-boot:run`

这意味着后端使用的是 Spring Boot 的开发启动方式，不是先打 jar 再手动启动。

#### `scripts/run-frontend-dev.cmd`

它现在会：

1. 优先寻找 `%USERPROFILE%\tools\node-v14.16.0-win-x64\npm.cmd`
2. 找不到就退回系统里的 `npm.cmd`
3. 切到 `frontend/`
4. 执行 `npm run serve --scripts-prepend-node-path=true`

### 2.5 推送当前分支脚本

如果你只是想把当前分支推到远端，可以执行：

```powershell
.\scripts\push-current-branch.cmd
```

或者：

```powershell
powershell -ExecutionPolicy Bypass -File .\scripts\push-current-branch.ps1
```

这个脚本会：

1. 自动识别当前分支。
2. 检查当前分支是否已经配置 upstream。
3. 如果已有 upstream，执行 `git push`。
4. 如果没有 upstream，执行 `git push --set-upstream origin <当前分支>`。

它不会自动提交代码，也不会自动合并分支。

这里特意保留了 Node 14 优先的逻辑，因为当前 `frontend/package.json` 的 `engines` 明确写的是：

- Node: `14.16.0`
- npm: `6.14.11`

### 2.6 启动后的访问地址

- 前端开发服务: `http://127.0.0.1:6618`
- 后端开发服务: `http://127.0.0.1:8091`
- 后端健康检查: `http://127.0.0.1:8091/api/system/health`
- H2 Console: `http://127.0.0.1:8091/h2-console`

### 2.7 怎么停掉本地服务

执行：

```powershell
powershell -ExecutionPolicy Bypass -File .\scripts\stop-local.ps1
```

`stop-local.ps1` 会读取 `runtime-logs/local-dev/pids.json`，然后逐个 `Stop-Process -Force`。

### 3.3 前端本地包依赖修正

## 4. Docker Compose 一键部署

### 4.1 入口文件

本次新增的容器化文件有：

- `docker-compose.yml`
- `backend/Dockerfile`
- `backend/.dockerignore`
- `frontend/Dockerfile`
- `frontend/.dockerignore`
- `frontend/nginx/default.conf`
- `scripts/docker-up.cmd`
- `scripts/docker-up.ps1`
- `scripts/docker-down.cmd`
- `scripts/docker-down.ps1`

### 4.2 推荐执行方式

启动：

```powershell
powershell -ExecutionPolicy Bypass -File .\scripts\docker-up.ps1
```

Windows 里也可以直接用：

```powershell
.\scripts\docker-up.cmd
```

停止：

```powershell
powershell -ExecutionPolicy Bypass -File .\scripts\docker-down.ps1
```

Windows 里也可以直接用：

```powershell
.\scripts\docker-down.cmd
```

### 4.3 `docker-up.ps1` 做了什么

它只做一件事：

```powershell
docker compose -p pvms up --build -d
```

然后打印访问地址。

### 4.4 为什么脚本里用了 `-p pvms`

因为当前仓库目录包含中文，Docker Compose 在 Windows 上自动推导 project name 时可能失败。  
所以脚本和 `docker-compose.yml` 都统一使用固定项目名 `pvms`，避免出现：

- project name must not be empty

### 4.5 `docker-compose.yml` 的服务关系

它定义了两个服务：

#### `backend`

- build context: `./backend`
- image build file: `backend/Dockerfile`
- 容器内端口: `8091`
- 宿主机映射: `8091:8091`

#### `frontend`

- build context: `./frontend`
- image build file: `frontend/Dockerfile`
- 容器内端口: `80`
- 宿主机映射: `6618:80`
- 依赖 `backend`

### 4.6 后端镜像构建过程

`backend/Dockerfile` 是两阶段构建：

#### 第 1 阶段：`maven:3.9.9-eclipse-temurin-21`

做的事情：

1. 复制 `pom.xml`
2. 复制 `src/`
3. 执行 `mvn -q -DskipTests package`

产物：

- `target/pvms-backend-0.0.1-SNAPSHOT.jar`

#### 第 2 阶段：`eclipse-temurin:21-jre`

做的事情：

1. 把 jar 复制到 `/app/pvms-backend.jar`
2. 暴露 8091 端口
3. 用 `java -jar /app/pvms-backend.jar` 启动

### 4.7 前端镜像构建过程

`frontend/Dockerfile` 也是两阶段构建：

#### 第 1 阶段：`node:22-bullseye`

做的事情：

1. 复制 `package.json`、`package-lock.json`、`vue.config.js`、`babel.config.js`
2. 复制 `public/`、`src/`、`vendor/`
3. 执行 `npm install --no-audit --no-fund --legacy-peer-deps`
4. 执行 `npm run build`

这里选择 Node 22 做容器构建器，是为了和当前 `package-lock.json`、历史依赖兼容；本地开发脚本仍然优先使用 Node 14。

#### 第 2 阶段：`nginx:1.27-alpine`

做的事情：

1. 用 `frontend/nginx/default.conf` 覆盖 Nginx 默认站点配置
2. 把前端构建产物复制到 `/usr/share/nginx/html/pvms`
3. 暴露 80 端口

### 4.8 Nginx 配置具体做了什么

`frontend/nginx/default.conf` 解决了 3 件事：

1. 访问 `/` 时自动跳转到 `/pvms/`
2. 访问 `/pvms/*` 时用 `try_files` 支持 Vue Router 的 history 路由刷新
3. 访问 `/api/*` 时反向代理到 `http://backend:8091`

这和 `frontend/vue.config.js` 的生产配置是对应的：

- `publicPath` 在生产环境下是 `/pvms/`

### 4.9 容器部署后的访问方式

- 前端: `http://127.0.0.1:6618/pvms/`
- 后端: `http://127.0.0.1:8091`
- 后端健康检查: `http://127.0.0.1:8091/api/system/health`

## 5. 这次实际验证到了哪一步

### 已验证

- `docker compose -p pvms config` 可以正确展开
- `mvn test` 通过
- `npm run build` 可以完成构建

### 未完全验证

- `docker compose build`
- `docker compose up -d`

原因不是配置文件本身报错，而是本机当时 Docker Desktop Linux Engine 没有运行，报错是：

```text
open //./pipe/dockerDesktopLinuxEngine: The system cannot find the file specified
```

所以如果你看到 compose 无法启动，第一件要做的事不是改 YAML，而是先启动 Docker Desktop。

## 6. 运行与部署时最容易踩坑的地方

### 6.1 H2 是内存库

后端每次重启都会重新执行：

- `backend/src/main/resources/schema.sql`
- `backend/src/main/resources/data.sql`

所以：

- 当前环境适合开发、联调、演示
- 不适合保存真实业务数据

### 6.2 前端是 Vue Router history 模式

开发环境靠 `frontend/vue.config.js` 里的：

- `historyApiFallback: true`

容器环境靠 `frontend/nginx/default.conf` 里的：

- `try_files $uri $uri/ /pvms/index.html;`

如果未来改部署方式，这两层必须同时考虑。

### 6.3 前端请求入口不是 axios 直写

所有页面都通过 `frontend/src/api/pvms.js` 调 `request()`，而不是每个页面自己拼 `axios`。  
真正决定“走宿主 / 走后端 / 走 mock”的地方在：

- `frontend/src/shared/host/bridge.js`

当前默认是：

- `VUE_APP_REQUEST_MODE` 不设时，优先走后端
- 只有显式设置 `mock` 时才走前端 mock

### 6.4 Docker 和本地开发不是同一条前端链路

- 本地开发：`npm run serve`，有 dev server、proxy、热更新
- Docker 部署：先 `npm run build`，再由 Nginx 提供静态资源和反向代理

所以本地能跑，不等于容器一定能跑；容器能跑，也不代表热更新链路没问题。
