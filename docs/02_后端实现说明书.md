# 后端实现说明书

这份文档的目标不是讲“后端有什么目录”，而是讲：

1. 一个接口从 URL 进来之后，会落到哪个 controller。
2. controller 调了哪个 service。
3. service 是查表、做汇总，还是直接生成 mock 数据。
4. 最终返回的 `data` 大概长什么样。

## 1. 后端总体结构

### 1.1 启动入口

- 启动类：`backend/src/main/java/cn/techstar/pvms/backend/PvmsBackendApplication.java`
- 技术栈：Spring Boot 3.2.5 + Spring Web + Validation + JdbcClient + H2

### 1.2 公共响应壳

所有控制器最终都返回 `ApiResponse<T>`：

- 文件：`backend/src/main/java/cn/techstar/pvms/backend/common/ApiResponse.java`

返回结构固定为：

```json
{
  "code": 0,
  "message": "success",
  "data": {}
}
```

所以你在看任何 controller 时，都应该把关注点放在 `data` 内部结构。

### 1.3 当前后端不是单一架构

当前后端分成两类：

#### A. H2 + JdbcClient + 计算型 service

这类模块会真的查表、做聚合、做曲线插值或衍生指标计算。

当前属于这一类的有：

- `dashboard` 的地图链路
- `productionmonitor`
- `stationarchive`

#### B. 纯 mock service

这类模块虽然也有 controller 和 test，但 service 内部主要是：

- 固定常量
- 内存列表
- 正弦/余弦曲线
- `Math.random()`

当前属于这一类的有：

- `forecast`
- `strategy`
- `devicealarm`
- `stations`
- `adjustable`
- `dashboard` 的部分旧接口

## 2. 启动与初始化链路

### 2.1 配置文件

- 文件：`backend/src/main/resources/application.yml`

关键配置：

- 端口：`8091`
- 数据源：H2 内存库 `jdbc:h2:mem:pvms`
- H2 Console：`/h2-console`
- SQL 初始化：`spring.sql.init.mode=always`

### 2.2 启动时会发生什么

后端启动时，Spring Boot 会：

1. 创建 H2 内存数据库。
2. 执行 `schema.sql` 建表。
3. 执行 `data.sql` 灌入种子数据。
4. 装配 controller / service / repository。

所以当前数据库里的数据不是业务用户录入的，而是启动时按 SQL 自动生成的。

4. 组装 3 类内容：
   - `filters`
   - `summary`
   - `stations`
5. `summary` 不是数据库字段，而是按过滤后的站点列表重新计数得到的。

返回 `data` 结构重点：

- `filters.statusOptions`
- `filters.regionOptions`
- `filters.capacityOptions`

- 告警表：`id, station_id, event_time, level, level_label, device_name, alarm_type, description, status, owner, suggestion`
- 站点表：`name`

服务层处理：

1. `summary()` 先全表统计 `critical / major / minor / hint`
2. `findRecent(level, stationId)` 再做条件过滤
3. 返回：
   - `summary`
   - `items`

### `GET /api/pvms/dashboard/vpp-node-status`

查询表：

- `dashboard_vpp_node_snapshot`
- `dashboard_station`
- `dashboard_station_status_snapshot`

服务层处理：

1. 取默认 VPP 节点快照。
2. 再把全部站点读出来。
3. 计算：
   - `availableCapacityMw`
   - `onlineStations`
   - `totalStations`
   - `adjustableRangeMw.min/max`

这里的 `availableCapacityMw` 不是表字段，而是 service 根据站点状态重新算出来的。

## 3.3 production-monitor 模块

这是当前最完整的后端模块之一。

### 控制器

- 路由前缀：`/api/pvms/production-monitor`

### Service

- 主 service：`ProductionMonitorDataService`
- 聚合器：`ProductionMonitorSeriesAggregator`

### Repository

- `ProductionMonitorResourceUnitRepository`
- `ProductionMonitorStationSnapshotRepository`
- `ProductionMonitorWeatherSnapshotRepository`
- `ProductionMonitorCurveRepository`
- `ProductionMonitorDispatchRecordRepository`

### 3.3.1 `GET /meta`

查询表：

- `pm_resource_unit`
- `pm_station`
- `pm_station_realtime_snapshot`

核心字段：

- 资源单元：`id, name, region, city, status, cluster_radius_km, dispatch_mode, strategy_label`
- 成员站点：`resource_unit_id, capacity_mw, status`

服务层处理：

1. 查所有资源单元。
2. 查所有站点快照并按 `resourceUnitId` 分组。
3. 生成：
   - 默认资源单元
   - 区域筛选项
   - 每个资源单元的元信息

### 3.3.2 `GET /overview?resourceUnitId=...`

查询表：

- `pm_resource_unit`
- `pm_station`
- `pm_station_realtime_snapshot`
- `pm_weather_snapshot`
- `pm_station_curve_15m`

核心字段：

- 资源单元告警信息：`latest_alarm_title`, `latest_alarm_time`, `alarm_total`, `alarm_critical`, `alarm_major`, `alarm_minor`
- 站点快照：`capacity_mw`, `status`, `online_rate`, `realtime_power_kw`, `load_kw`
- 天气：`weather, cloudiness, temperature, irradiance, humidity, wind_speed, conclusion`
- 曲线：`pv_power_kw, forecast_power_kw`

服务层处理重点：

1. `resolveContext()` 一次性把资源单元、站点、天气、曲线都取齐。
2. `seriesAggregator.aggregate()` 把 15 分钟点聚成统一时序点。
3. 计算这些派生指标：
   - `dispatchableCapacityMw`
   - `realtimePowerMw`
   - `todayEnergyMwh`
   - `upRegulationMw`
   - `downRegulationMw`
   - `onlineRate`
   - `accuracy`
4. 组装：
   - `info`
   - `kpis`
   - `memberStations`
   - `weatherBrief`
   - `alarmBrief`
   - `summaryTable`

### 3.3.3 `GET /output?resourceUnitId=...&granularity=15m|30m|60m`

查询表：

- `pm_station_curve_15m`
- `pm_station`
- `pm_weather_snapshot`

核心字段：

- 曲线：`load_kw, pv_power_kw, forecast_power_kw, baseline_power_kw, irradiance, temperature`
- 站点：`weight, status, sort_index`

服务层处理重点：

1. `ProductionMonitorSeriesAggregator.resolveBucketSize()` 把粒度换算为 15/30/60 分钟桶。
2. 按桶聚合出：
   - `actual`
   - `forecast`
   - `baseline`
   - `irradiance`
3. 再计算：
   - 峰值出力
   - 当前偏差率
   - 贡献排名
   - 表格行数据

### 3.3.4 `GET /load?resourceUnitId=&date=&granularity=`

查询表：

- `pm_station`
- `pm_station_realtime_snapshot`
- `pm_station_curve_15m`

这里特别要注意：

- 控制器接收 `resourceUnitId`
- 但当前 service 实现里 `getLoad()` 实际上按 `loadVisible = true` 的站点全局汇总，并没有再按资源单元过滤

服务层处理重点：

1. 筛出 `loadVisible = true` 的站点。
2. 查询这些站点在指定日期的曲线。
3. 对每个站点生成：
   - `realtimePowerKw`
   - `loadKw`
   - `adjustableKw`
   - `maxRampRate`
   - `gridInteraction`
4. 总结为：
   - `totalLoadMw`
   - `totalPvOutputMw`
   - `totalAdjustableMw`
   - `avgRampRate`
   - `stationCount`
   - `onlineCount`

### 3.3.5 `GET /dispatch?resourceUnitId=...`

查询表：

- `pm_dispatch_record`
- `pm_station`
- `pm_station_realtime_snapshot`
- `pm_weather_snapshot`

核心字段：

- 调度记录：`issued_at, command_type, target_power_mw, actual_power_mw, response_seconds, status, deviation_reason`

服务层处理重点：

1. 查调度记录。
2. 按小时分组。
3. 计算：
   - 指令数
   - 成功数
   - 成功率
   - 平均响应时长
   - 今日可调用量
4. 额外组装：
   - `executionTrend`
   - `riskHints`
   - `records`

### 3.3.6 `GET /weather?resourceUnitId=...`

查询表：

- `pm_weather_snapshot`
- `pm_station`

服务层处理重点：

1. 从天气快照里取当前天气。
2. 基于当前温度、辐照度、风速拼出一个“简化趋势”。
3. 生成 `impactTable`，说明天气对出力的影响建议。

   - 负荷
   - 可调空间
   - 当日发电量
   - 预测准确率

### 3.4.5 `GET /api/pvms/station-archive/station-realtime`

参数：

- `stationId`
- `metric`
- `date`
- `granularity`

这一条接口实际是一条“多态接口”。  
`metric` 不同，返回的 `data` 结构也不同。

#### `metric=adjustable`

返回：

- `times`
- `series`
- `kpis`
- `stationKpis`
- `monthlyStats`
- `fields`

数据来源：

- `sa_station`
- `sa_station_curve_15m`

#### `metric=pv-output`

返回：

- `times`
- `series`

数据来源：

- `sa_station_curve_15m`
- `sa_inverter`

服务层会把站级出力按逆变器权重拆成多条线。

#### `metric=load`

返回：

- `times`
- `series`

这里负荷不是分项表直接查出来的，而是 service 按比例把总负荷拆成：

- 空调
- 照明
- 动力设备

#### `metric=forecast`

返回：

- `times`
- `series`

直接使用：

- `forecast_day_ahead_kw`
- `forecast_ultra_short_kw`
- `pv_output_kw`

### 3.4.6 `GET /api/pvms/station-archive/station-adjustable`

查询表：

- `sa_station_curve_15m`

服务层处理：

1. 计算 `load - pvOutput`
2. 得到：
   - `currentPowerKw`
   - `maxAdjustableKw`
   - `minAdjustableKw`
   - `adjustableRangeKw`
   - `utilizationRate`
   - `periods`

### 3.4.7 `GET /api/pvms/station-archive/station-strategy`

查询表：

- `sa_station_strategy`

服务层处理：

1. 读取当前站点策略。
2. 不查执行日志表。
3. `executionLogs` 是 service 按策略状态动态拼出来的演示日志。

### 3.4.8 `GET /api/pvms/station-archive/inverter-realtime`

查询表：

- `sa_inverter`
- `sa_station`
- `sa_station_curve_15m`
- `sa_inverter_alarm`

服务层处理重点：

1. 找逆变器。
2. 找所属站点。
3. 取站级功率曲线。
4. 按逆变器额定功率和状态因子把站级曲线分摊成逆变器曲线。
5. 额外组装：
   - `topology`
   - `alarms`
   - `deviceInfo`

### 3.4.9 时序插值逻辑

`StationArchiveSeriesService` 负责两件核心事：

1. `15min` 原始曲线直接按 96 个点返回。
2. `1min` 粒度时，对 15 分钟点做线性插值，扩成 1440 个点。

这也是为什么前端切到 `1min` 后数据量会大幅增加。

## 4. 维护时最重要的判断题

### 4.1 先判断这个接口属于哪一类

你改接口之前先回答：

- 这个接口是 H2/JDBC 链路，还是 mock 链路？

如果是 H2/JDBC：

- 先看 repository SQL
- 再看 service 派生计算

如果是 mock：

- 先看 service 内部常量
- 再看前端到底依赖哪些字段

### 4.2 当前最值得优先真实化的后端模块

如果后续继续演进，建议优先顺序：

1. `forecast`
2. `strategy`
3. `devicealarm`
4. `stations`
5. `adjustable`

原因很简单：这些模块前端都已经有页面，但后端还主要停留在演示态。
