# 前端实现说明书

这份文档的目标是把前端从黑盒拆开。你应该能顺着“页面 -> 组件 -> API -> bridge -> 后端接口”一路反查到代码。

---

## 1. 启动与请求总链路

## 1.1 入口文件：`frontend/src/main.js`

前端启动时依次做这些事：

1. 引入 `frontend/src/styles/index.less`
2. 初始化高德配置：`applyAmapSecurityConfig()`
3. 初始化宿主桥接：`initializeHostBridge(props)`
4. 创建路由：`createRouter(...)`
5. 创建 Store：`createStore()`
6. 执行 `store.dispatch('initApp', props)`
7. 挂载 `App.vue`

真实链路可以记成：

`main.js -> bridge.js -> router/index.js -> store/index.js -> App.vue`

## 1.2 路由注册：`frontend/src/router/index.js`

当前所有业务页面都挂在 `ShellLayout.vue` 下。主要路由如下：

| 路由 | 页面组件 | 说明 |
| --- | --- | --- |
| `/dashboard/overview` | `modules/dashboard/pages/OverviewPage.vue` | 首页综合监控 |
| `/production-monitor/overview` | `modules/production-monitor/pages/ProductionMonitorPage.vue` | 资源总览 |
| `/production-monitor/output` | 同上 | 出力分析 |
| `/production-monitor/dispatch` | 同上 | 调度执行 |
| `/production-monitor/station` | 同上 | 电站监控 |
| `/production-monitor/load` | 同上 | 负荷与出力 |
| `/forecast/overview` | `modules/forecast/pages/ForecastPage.vue` | 预测总览 |
| `/forecast/adjustable` | 同上 | 可调能力 |
| `/forecast/accuracy` | 同上 | 精度评估 |
| `/strategy/list` | `modules/strategy/pages/StrategyPage.vue` | 策略列表 |
| `/strategy/config` | 同上 | 策略配置 |
| `/strategy/revenue` | 同上 | 收益分析 |
| `/devices/monitor` | `modules/devices/pages/DeviceMonitorPage.vue` | 设备监测 |
| `/alarms/center` | `modules/alarms/pages/AlarmCenterPage.vue` | 告警中心 |

`meta.title / meta.section / meta.summary / meta.viewKey` 很重要，它决定了：

- 顶部页头文案
- 左侧菜单高亮
- 一个 Page 组件内部具体渲染哪个子视图

## 1.3 Store：`frontend/src/store/index.js`

Store 比较轻，主要管应用级信息：

- `appName`
- `appDescription`
- `menus`
- `runtime`

它的数据来自两处：

- `frontend/src/settings.js`
- `frontend/src/shared/host/bridge.js` 的 `buildRuntimeContext()`

所以：

- 菜单顺序、图标、描述去 `settings.js`
- 当前租户、用户名、是否宿主模式去 `bridge.js`

## 1.4 请求桥：`frontend/src/shared/host/bridge.js`

这是前端最关键的非页面文件。所有 `src/api/pvms.js` 方法最终都走这里。

当前请求顺序：

1. 有宿主 `window._tsx.apirequest` 时，走宿主
2. 否则默认走后端 `/api`
3. 只有显式 `VUE_APP_REQUEST_MODE=mock` 才走 `mockHandlers`

这说明当前工作区已经不是“开发环境默认只走 mock”的旧模式了。

## 1.5 API 封装：`frontend/src/api/pvms.js`

页面层一般不直接写 URL，而是通过这个文件统一封装接口方法，例如：

- `fetchProductionMonitorOverview()`
- `fetchDashboardStationsGeo()`
- `fetchForecastMeta()`
- `fetchStrategyRevenueSummary()`

排查接口问题时建议顺序：

1. 页面组件
2. `pvms.js`
3. `bridge.js`
4. 后端 controller/service

## 2. 共享布局与样式体系

## 2.1 布局壳：`frontend/src/layouts/ShellLayout.vue`

这是所有页面共同的外框，包含：

- 左侧品牌区
- 左侧租户区
- 左侧菜单
- 顶部标题区
- 顶部用户区
- 内容区 `router-view`

它从 Store 和 `$route.meta` 取数据：

- `appName`
- `appDescription`
- `menus`
- `runtime.tenantName`
- `runtime.userName`
- `currentRouteMeta`

如果你想改整体侧边栏、页头或者主框架背景，优先改这个文件。

## 2.2 全局样式层

### `frontend/src/styles/tokens.less`

这里定义全局主题变量，例如：

- `--pvms-primary`
- `--pvms-bg-body`
- `--pvms-panel`
- `--pvms-text-primary`
- `--pvms-shadow-soft`

### `frontend/src/styles/element-overrides.less`

这里改 Element UI 的全局样式，包括：

- `el-table`
- `el-input`
- `el-select`
- `el-button`
- `el-pagination`

### `frontend/src/styles/index.less`

这里放跨页面通用布局类，例如：

- `.pv-page`
- `.pv-card-grid`
- `.pv-split-grid`
- `.pv-filter-grid`
- `.pv-table-shell`

结论：

- 主题色问题先看 `tokens.less`
- 所有表格/输入框统一长相先看 `element-overrides.less`
- 栅格和通用页面骨架先看 `index.less`

## 2.3 共享组件

### `AppPageHero.vue`

页面头图组件，负责：

- `eyebrow`
- `title`
- `description`
- 右上角 `meta` 插槽

### `AppSectionCard.vue`

统一的卡片容器组件，负责：

- 标题
- 副标题
- 右上角扩展位

### `AppMetricCard.vue`

统一 KPI 卡片，渲染：

- `title`
- `value`
- `unit`
- `helper`
- `icon`
- `accent`

它的颜色不是页面自己算的，而是通过 `resolveAccentClass()` 映射到卡片强调色类。

## 3. 模块级实现

## 3.1 首页综合监控：`/dashboard/overview`

页面文件：

- `frontend/src/modules/dashboard/pages/OverviewPage.vue`

页面由这些组件组成：

- `AppPageHero`
- `DashboardMapCard`
- `DashboardKpiBoard`
- `DashboardPowerCurve`
- `DashboardStationRanking`
- `DashboardAlarmFeed`
- `DashboardAlarmDrawer`

页面加载这些接口：

- `fetchDashboardStationsGeo()`
- `fetchDashboardKpiSummary()`
- `fetchDashboardPowerCurve()`
- `fetchDashboardStationRanking()`
- `fetchDashboardAlarmFeed()`
- `fetchAdjustableCapacity()`
- `fetchVppNodeStatus()`

核心状态变量：

- `mapPayload`
- `kpiPayload`
- `curvePayload`
- `rankingPayload`
- `alarmPayload`
- `adjustablePayload`
- `vppNodePayload`
- `mapFilters`
- `selectedStationId`

渲染关系：

- `mapPayload` 给地图组件画点位
- `kpiPayload` 给 KPI 面板
- `curvePayload` 给曲线组件
- `rankingPayload` 给排行组件
- `alarmPayload` 给告警列表和抽屉

当前最重要的事实：

- 地图点位、最近告警、VPP 节点是走真实后端接口
- KPI、功率曲线、排行榜、可调容量仍是旧 mock 链路

所以这个页面是“混合态”，不是全量真实接口。

样式来源：

- `OverviewPage.vue` 自己的 `scoped less`
- 各子组件自己的局部样式
- 全局卡片/表格栅格样式来自 `styles/index.less`

## 3.2 生产监控：`/production-monitor/*`

页面文件：

- `frontend/src/modules/production-monitor/pages/ProductionMonitorPage.vue`

这个页面一套壳承接五个路由，通过 `$route.meta.viewKey` 切子视图：

- `overview`
- `output`
- `dispatch`
- `station`
- `load`

主要组件：

- `ProductionMonitorHero`
- `ProductionMonitorTabNav`
- `ProductionMonitorFilterBar`
- `ProductionOverviewView`
- `ProductionOutputView`
- `ProductionDispatchView`
- `ProductionLoadView`
- `StationTreePanel`
- `StationDataPanel`

主要接口：

- `fetchProductionMonitorMeta()`
- `fetchProductionMonitorOverview()`
- `fetchProductionMonitorOutput()`
- `fetchProductionMonitorLoad()`
- `fetchProductionMonitorDispatch()`
- `fetchStationTree()`
- `fetchStationArchiveCompanyOverview()`
- `fetchStationArchiveInverterRealtime()`
- `fetchStationArchiveRealtime()`
- `fetchStationArchiveStrategy()`

理解这个页面时要分两种模式：

### A. 资源单元模式

对应 `overview/output/dispatch/load`。  
这时页面核心状态围绕 `resourceUnitId`，数据来自 `production-monitor` 后端模块。

### B. 电站模式

对应 `station`。  
左边是 `StationTreePanel` 树，右边是 `StationDataPanel` 详情。虽然页面还叫 Production Monitor，但右侧大部分数据实际来自 `station-archive` 接口。

额外机制：

- 路由 query 会同步当前 `resourceUnitId`
- 电站模式带 refresh timer，会刷新实时面板

样式来源：

- 页面壳：`ProductionMonitorPage.vue`
- 子视图：`components/views/*.vue`
- 电站树和右侧面板：各自组件文件内 `scoped less`

## 3.3 预测分析：`/forecast/*`

页面文件：

- `frontend/src/modules/forecast/pages/ForecastPage.vue`

主要组件：

- `ForecastHero`
- `ForecastTabNav`
- `ForecastFilterBar`
- `ForecastOverviewView`
- `ForecastAdjustableView`
- `ForecastAccuracyView`

接口：

- `fetchForecastMeta()`
- `fetchForecastOverview()`
- `fetchForecastAdjustable()`
- `fetchForecastAccuracy()`
- `fetchForecastComparison()`
- `fetchForecastDeviationHeatmap()`

这个页面的关键点不是“直接把接口数据扔给子组件”，而是页面层会先做一次组装。

例如 `overview` 视图会把：

- `overview`
- `comparison`
- `deviation-heatmap`

三份数据揉成一个 `viewData` 再传给 `ForecastOverviewView.vue`。

这就是为什么子视图里经常出现这些字段：

- `viewData.kpis`
- `viewData.comparison`
- `viewData.heatmap`
- `viewData.stationTable`

而后端原始响应却不是完全同名。

当前风险：

- 前端页面期望的结构偏页面友好
- 后端 `ForecastMockService` 当前保护的结构偏 `kpi/accuracyTrend/forecastVsActual/matrix`

所以 forecast 当前有明确的前后端契约漂移风险。

## 4. 前端排查路线图

## 4.1 看到页面某个字段不对

按这个顺序查：

1. 页面组件里这个字段来自哪个状态变量
2. 状态变量由哪个 API 方法赋值
3. `pvms.js` 实际调了哪个 URL
4. `bridge.js` 这次请求是走宿主、后端，还是 mock
5. 最后再去后端 controller/service 看真实数据源

## 4.2 看到样式不对

先判断是哪个层级：

- 全局主题：`tokens.less`
- 所有 Element 组件：`element-overrides.less`
- 页面通用骨架：`index.less`
- 单个共享组件：对应组件自己的 `scoped less`
- 单个业务页面：对应页面文件自己的 `scoped less`

## 4.3 当前最重要的前端事实

1. 当前默认请求已经是后端优先，不是 mock 优先。
2. 首页 `dashboard` 仍然处于“部分真实 + 部分 mock”混合态。
3. `production-monitor` 和 `station-archive` 是当前最适合反查真实链路的前端模块。
4. `forecast` 和 `strategy` 都存在页面层二次装配数据，因此字段名不能直接拿来等同后端原始字段名。
