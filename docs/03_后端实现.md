## 3. 配置与启动

### 3.1 启动类

```java
@SpringBootApplication
@MapperScan("cn.techstar.pvms.backend")
public class PvmsBackendApplication {
    public static void main(String[] args) {
        SpringApplication.run(PvmsBackendApplication.class, args);
    }
}
```

`@MapperScan` 自动扫描所有带 `@Mapper` 注解的接口，无需逐个配置。

### 3.2 生产配置（application.yml）

```yaml
server:
  port: 8091

spring:
  datasource:
    driver-class-name: com.mysql.cj.jdbc.Driver
    url: jdbc:mysql://localhost:3306/pvms?useUnicode=true&characterEncoding=utf-8
         &useSSL=false&serverTimezone=Asia/Shanghai&allowPublicKeyRetrieval=true
    username: root
    password: "123456"

mybatis:
  mapper-locations: classpath:mapper/**/*.xml
  configuration:
    map-underscore-to-camel-case: true
    arg-name-based-constructor-auto-mapping: true
```

### 3.3 测试配置（application-test.yml）

```yaml
spring:
  datasource:
    driver-class-name: org.h2.Driver
    url: jdbc:h2:mem:pvms_test;MODE=MySQL;DB_CLOSE_DELAY=-1
    username: sa
    password:
  sql:
    init:
      mode: always

      data-locations: classpath:data.sql
```

测试使用 H2 内存库（MySQL 兼容模式），自动加载 schema.sql + data.sql。

### 3.4 编译配置

`pom.xml` 中 `maven-compiler-plugin` 启用了 `-parameters` 选项，使 MyBatis 能够通过反射获取方法参数名，支持 Java record 的构造器自动映射。

## 4. 分层架构与设计决策

### 4.1 三层分工

| 层 | 职责 | 实现方式 |
|---|---|---|
| **Controller** | 接收 HTTP 请求，提取参数，返回统一响应 | `@RestController` + `ApiResponse<T>` |
| **Service** | 业务逻辑：聚合多 Mapper 数据、计算 KPI、过滤排序、组装 JSON | `@Service`，返回 `Map<String, Object>` |
| **Mapper** | 数据访问：单表或简单联查的 SQL 操作 | `@Mapper` 接口 + XML |

### 4.2 Mapper 设计模式

所有 Mapper 使用以下模式：

```java
@Mapper
public interface SomeMapper {
    List<SomeRow> findAll();
    SomeRow findById(@Param("id") String id);

    record SomeRow(String id, String name, ...) {}
}
```

**关键特征**：
- 结果类型使用 **Java record**（不可变、简洁、自带 equals/hashCode）
- Record 定义为 Mapper 接口的**内部类型**，与 SQL 映射紧密绑定
- XML 中使用 `resultType` 自动映射，配合 `map-underscore-to-camel-case: true`
- 列别名必须与 record 字段名的下划线形式匹配（如字段 `stationId` 对应列 `station_id`）

### 4.3 为什么 Service 返回 Map

当前所有 Service 方法返回 `Map<String, Object>` 而非 DTO。这是一个有意识的折衷：
- **优点**：开发期灵活，前后端字段快速迭代，无需频繁创建/修改 DTO 类
- **缺点**：无编译期类型保护，重构时不易发现字段变更
- **后续建议**：在接口契约稳定后，逐步为核心接口引入 DTO

### 4.4 统一响应体

```java
public record ApiResponse<T>(int code, String message, T data) {
    public static <T> ApiResponse<T> success(T data) {
        return new ApiResponse<>(0, "success", data);
    }
}
```

所有接口返回格式统一为 `{ "code": 0, "message": "success", "data": {...} }`。

## 6. M01 综合监控中心（Dashboard）

### 6.1 模块概览

| 组件 | 类名 |
|---|---|
| Controller × 3 | `DashboardController` / `AlarmController` / `WeatherController` |
| Service × 1 | `DashboardMapDataService` |
| Mapper × 3 | `DashboardStationGeoMapper` / `DashboardAlarmSnapshotMapper` / `DashboardVppNodeSnapshotMapper` |
| 数据表 × 4 | `dashboard_station` / `dashboard_station_status_snapshot` / `dashboard_alarm_snapshot` / `dashboard_vpp_node_snapshot` |

### 6.2 API 接口

#### GET `/api/pvms/dashboard/stations-geo`

获取所有电站的 GIS 数据，支持筛选。

| 参数 | 类型 | 必填 | 说明 |
|---|---|---|---|
| status | String | 否 | 电站状态过滤 |
| region | String | 否 | 区域过滤 |
| capacityRange | String | 否 | 容量范围（lt3000 / 3000to5000 / gte5000） |

**业务逻辑**：
1. 从 `DashboardStationGeoMapper.findAll()` 查出所有电站（联查 station + status_snapshot）
2. 按 status、region、capacityRange 进行内存过滤
3. 构建筛选选项列表（状态/区域/容量）
4. 构建摘要统计（各状态的数量）
5. 组装每个电站的完整 JSON（含状态标签、颜色、经纬度、KPI 等）

#### GET `/api/pvms/dashboard/kpi-summary`

| 参数 | 说明 |
|---|---|
| stationId（可选） | 指定电站，为空则统计全系统 |

返回装机容量、实时功率、今日发电量、今日收益、等效利用小时、可用率、CO2 减排、功率偏差。

#### GET `/api/pvms/dashboard/power-curve`

| 参数 | 说明 |
|---|---|
| stationId（可选） | 指定电站 |
| date（可选） | 日期，默认当天 |

返回 96 个数据点（15 分钟粒度）的实际出力和计划出力曲线。曲线基于正弦函数和日期种子生成。

#### GET `/api/pvms/dashboard/station-ranking`

| 参数 | 说明 |
|---|---|
| metric（可选） | 排名维度：energy / hours / pr |

返回所有电站按指定指标的降序排名。

#### GET `/api/pvms/dashboard/overview`

汇总接口，返回总装机、总功率、发电量等 KPI 卡片 + 告警摘要 + 最近告警列表。

#### GET `/api/pvms/dashboard/vpp-node-status`

返回 VPP 节点的总容量、可用容量、在线电站数、可调范围、最后心跳时间。

#### GET `/api/pvms/alarms/recent`

| 参数 | 说明 |
|---|---|
| level（可选） | 告警级别过滤 |
| stationId（可选） | 电站过滤 |

返回告警汇总（各级别数量）+ 告警明细列表。

#### GET `/api/pvms/weather/current`

| 参数 | 说明 |
|---|---|
| stationId（可选） | 电站，用于匹配气象站 |

返回天气、温度、辐照度等气象数据（从 `pm_weather_snapshot` 获取）。

## 9. M02c 电站档案（Station Archive）

### 9.1 模块概览

| 组件 | 类名 |
|---|---|
| Controller × 1 | `StationArchiveController` |
| Service × 2 | `StationArchiveDataService` / `StationArchiveSeriesService` |
| Mapper × 6 | Company / Station / Curve / Inverter / InverterAlarm / Strategy |
| 数据表 × 7 | `sa_company` / `sa_station` / `sa_station_curve_15m` / `sa_inverter` / `sa_inverter_alarm` / `sa_station_strategy` |

### 9.2 API 接口

#### GET `/api/pvms/station-tree`

返回公司→电站→逆变器三层树结构。支持按状态过滤。

**业务逻辑**：
1. 查询所有公司、电站、逆变器
2. 按层级组装树节点，每个节点包含 id、label、nodeType、children
3. 若传入 status 过滤，则只保留匹配状态的电站及其父公司

#### GET `/api/pvms/station-archive/company-overview`

| 参数 | 说明 |
|---|---|
| companyId | 公司 ID |

返回公司下所有电站的列表（容量、状态、并网状态、数据质量等级）。

#### GET `/api/pvms/station-archive/resource-overview`

| 参数 | 说明 |
|---|---|
| stationId | 电站 ID |

返回电站的详细基础信息。

#### GET `/api/pvms/station-archive/station-realtime`

| 参数 | 说明 |
|---|---|
| stationId | 电站 ID |
| metric | 数据维度：adjustable / pv-output / load / forecast |
| date | 日期 |
| granularity | 粒度：1min / 15min / 1h |

**业务逻辑**：
1. 查询电站基础信息和 KPI
2. 根据 metric 参数从 `sa_station_curve_15m` 提取对应的曲线字段
3. 如果粒度为 1min，使用 `StationArchiveSeriesService` 对 15min 数据做线性插值
4. 如果粒度为 1h，对 15min 数据做 4 点平均聚合

#### GET `/api/pvms/station-archive/station-adjustable`

返回电站的可调空间数据，基于预测分析模块的 `fc_adjustable_window` 表。

#### GET `/api/pvms/station-archive/station-strategy`

返回电站当前执行的策略，从 `sa_station_strategy` 表读取。

#### GET `/api/pvms/station-archive/inverter-realtime`

返回逆变器的完整台账信息和告警记录。

### 11.3 写操作 API

| 接口 | HTTP | 说明 |
|---|---|---|
| `/api/pvms/strategy/create` | POST | 创建策略，接收 StrategyRequest |
| `/api/pvms/strategy/batch-create` | POST | 批量创建，接收 StrategyBatchRequest |
| `/api/pvms/strategy/submit` | POST | 提交审批（id 参数） |
| `/api/pvms/strategy/terminate` | POST | 终止策略 |
| `/api/pvms/strategy/batch-submit` | POST | 批量提交 |
| `/api/pvms/strategy/batch-delete` | POST | 批量删除 |

注意：当前写操作在 Service 层**不实际写入数据库**，而是返回模拟的成功响应。这是因为系统仍处于联调准备阶段，写操作的真实持久化待后续实现。

### 11.4 模拟与收益 API

#### POST `/api/pvms/strategy/simulate`

接收 `StrategyRequest`，通过 `StrategySimulationService` 计算模拟收益：
- 预估日收益 = 目标功率 × 执行时长 × 平均电价 × 效率系数
- 置信区间 = 预估收益 × [0.75, 1.15]
- 成功概率基于策略类型映射（削峰填谷 92%、需求响应 85%、调频辅助 78%）

#### GET `/api/pvms/strategy/revenue/summary`

返回策略收益汇总：总预估/实际收益、偏差率，按策略拆分。

#### GET `/api/pvms/strategy/revenue/detail`

| 参数 | 说明 |
|---|---|
| startDate / endDate | 日期范围 |

返回日粒度的收益明细：预估收益、实际收益、削峰节省、响应奖励、偏差罚款。

#### GET `/api/pvms/strategy/compare`

| 参数 | 说明 |
|---|---|
| ids | 逗号分隔的策略 ID 列表 |

返回多个策略的收益对比数据。
