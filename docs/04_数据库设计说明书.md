# 数据库设计说明书

这份文档只讲当前仓库里真正存在的数据库设计，不讲“理想中应该怎么做”。

当前后端数据库的真实状态：

- 数据库类型：H2 内嵌内存数据库
- 初始化来源：`backend/src/main/resources/schema.sql` 和 `backend/src/main/resources/data.sql`
- 已数据库化模块：
  - dashboard 地图链路
  - production-monitor
  - station-archive
- 尚未数据库化模块：
  - forecast
  - strategy
  - devicealarm
  - stations
  - adjustable

---

## 1. 初始化方式

## 1.1 配置文件

文件：`backend/src/main/resources/application.yml`

关键配置：

- `server.port=8091`
- `spring.datasource.url=jdbc:h2:mem:pvms;MODE=MySQL;DB_CLOSE_DELAY=-1;DB_CLOSE_ON_EXIT=FALSE`
- `spring.h2.console.path=/h2-console`
- `spring.sql.init.mode=always`

## 1.2 启动时发生什么

Spring Boot 启动时会：

1. 创建 H2 内存库 `pvms`
2. 执行 `schema.sql` 建表
3. 执行 `data.sql` 灌入种子数据
4. 装配各模块的 `JdbcClient Repository`

因此当前数据库的特点很明确：

- 适合开发、联调、演示
- 不适合作为正式持久化存储
- 只要后端进程重启，数据就会重新初始化

## 3. dashboard 表设计

## 3.1 `dashboard_station`

用途：保存首页地图站点的静态主数据。

主键：

- `id`

核心字段：

- `id`
- `name`
- `resource_unit_id`
- `resource_unit_name`
- `region`
- `longitude`
- `latitude`
- `address`
- `capacity_kwp`

对应接口：

- `GET /api/pvms/dashboard/stations-geo`

说明：

这个表只存“站点是什么、在哪儿、属于谁”，不存动态功率和告警。

## 3.2 `dashboard_station_status_snapshot`

用途：保存站点的当前状态快照。

主键：

- `station_id`

外键：

- `station_id -> dashboard_station.id`

核心字段：

- `status`
- `realtime_power_kw`
- `today_energy_kwh`
- `today_revenue_cny`
- `availability`
- `health_grade`
- `snapshot_time`

服务层处理：

- 与 `dashboard_station` join
- 再按 `status/region/capacityRange` 过滤
- 最后重新汇总得到 `summary` 和 `stations`

## 3.3 `dashboard_alarm_snapshot`

用途：保存首页最近告警演示数据。

主键：

- `id`

外键：

- `station_id -> dashboard_station.id`

核心字段：

- `station_id`
- `event_time`
- `level`
- `level_label`
- `device_name`
- `alarm_type`
- `description`
- `status`
- `owner`
- `suggestion`

对应接口：

- `GET /api/pvms/alarms/recent`

服务层处理：

- 一次全表统计告警级别摘要
- 再根据 `level/stationId` 过滤输出 `items`

## 3.4 `dashboard_vpp_node_snapshot`

用途：保存首页 VPP 节点的静态快照。

主键：

- `node_id`

核心字段：

- `total_capacity_mw`
- `adjustable_min_mw`
- `adjustable_max_mw`
- `last_heartbeat`

对应接口：

- `GET /api/pvms/dashboard/vpp-node-status`

服务层处理：

- 先读取节点快照
- 再结合站点状态重算：
  - `availableCapacityMw`
  - `onlineStations`
  - `totalStations`
  - `adjustableRangeMw.min/max`

## 6. `data.sql` 的造数逻辑

当前造数不是“手工塞几条静态记录”，而是“主数据 + 规则生成时序数据”。

## 6.1 生产监控曲线

`pm_station_curve_15m` 使用 `SYSTEM_RANGE(0, 95)` 生成一天 96 个时隙。

生成思路：

- 用容量和权重决定量级
- 用正弦/余弦模拟日照与出力变化
- 同时构造：
  - 实际出力
  - 预测出力
  - 基线出力
  - 负荷
  - 辐照度
  - 温度

## 6.2 实时快照

`pm_station_realtime_snapshot` 不是独立维护，而是从某个固定时隙切片生成。

好处：

- 实时卡片能和时序曲线对上
- 不会出现“表格值和曲线值完全不一致”

## 6.3 调度记录

`pm_dispatch_record` 会参考资源单元规模和状态生成：

- 目标功率
- 实际功率
- 响应时间
- 偏差原因

所以调度记录并不是孤立的演示字符串。

## 6.4 电站档案曲线

`sa_station_curve_15m` 同样生成 96 个点，字段包括：

- `load_kw`
- `pv_output_kw`
- `forecast_day_ahead_kw`
- `forecast_ultra_short_kw`

这样电站监控右侧的负荷、实际出力、日前预测、超短期预测能互相关联。

## 6.5 逆变器

`sa_inverter` 的数量和额定功率是按站点容量分配出来的。  
站点越大，逆变器数量和能力也越大。

## 7. 当前数据库还没有覆盖的模块

`schema.sql` 里当前没有这些表：

- forecast 预测元数据表
- strategy 策略主表/收益表
- devicealarm 设备与告警持久化表
- stations 资源单元真实业务表
- adjustable 可调容量事实表

这意味着这些模块现在仍是 mock service，不是数据库链路。

## 8. 后续扩展建议

## 8.1 哪些字段适合继续直接存库

- 站点主数据
- 设备主数据
- 实际采集点
- 天气事实
- 调度记录
- 告警记录

## 8.2 哪些字段更适合服务层派生

- 在线率
- 成功率
- 可调空间
- KPI 汇总
- 预测偏差
- 站点排名

原因很简单：这些值是聚合结果，不应该在数据库里重复固化。

## 8.3 迁到正式数据库时的建议顺序

1. 先保留现有表分组思路
2. 把 H2 迁到 MySQL/PostgreSQL
3. 引入迁移工具（Flyway/Liquibase）
4. 再把 forecast、strategy、devicealarm 这些 mock 模块逐个落表
