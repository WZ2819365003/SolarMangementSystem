# 后端接口测试说明书

这份文档只讲后端接口测试。前端 Playwright 截图和页面回归留到后续单独整理。

目标：

1. 让接手人知道当前后端接口有哪些自动化保护。
2. 知道每个测试类覆盖了哪些 URL。
3. 知道入参是什么、核心断言是什么。
4. 知道每个测试在防什么回归。

---

## 1. 测试入口

## 1.1 技术栈

当前后端测试基于：

- Spring Boot Test
- MockMvc
- JUnit 5

所以这些测试不是浏览器测试，而是把 Spring Boot 应用拉起来后，对 controller 做 HTTP 级模拟请求。

## 1.2 全量执行

```powershell
cd backend
mvn.cmd test
```

## 1.3 单模块执行

例如只跑生产监控：

```powershell
cd backend
mvn.cmd -Dtest=ProductionMonitorControllerTest test
```

例如只跑电站档案：

```powershell
cd backend
mvn.cmd -Dtest=StationArchiveControllerTest test
```

## 1.4 本次实际验证结果

我在 `2026-03-31` 本地执行过：

```powershell
cd backend
mvn.cmd test
```

结果：

- `Tests run: 62`
- `Failures: 0`
- `Errors: 0`
- `Skipped: 0`
- `BUILD SUCCESS`

## 3.6 `StrategyControllerTest`

文件：

- `backend/src/test/java/cn/techstar/pvms/backend/module/strategy/controller/StrategyControllerTest.java`

覆盖接口：

- `GET /meta`
- `GET /tree`
- `GET /kpi`
- `GET /list`
- `GET /detail`
- `GET /electricity-price`
- `POST /create`
- `POST /batch-create`
- `POST /submit`
- `POST /terminate`
- `POST /batch-submit`
- `POST /simulate`
- `POST /batch-simulate`
- `GET /revenue/summary`
- `GET /revenue/detail`
- `GET /compare`

单策略创建请求体示例：

```json
{
  "name": "测试策略",
  "stationId": "ST-001",
  "type": "peak-shaving",
  "executeDate": "2026-03-30",
  "targetPowerKw": 1500,
  "powerUpperLimitKw": 1800,
  "powerLowerLimitKw": 1200
}
```

批量创建请求体示例：

```json
{
  "items": [
    { "name": "批量A", "stationId": "ST-001", "type": "peak-shaving", "targetPowerKw": 1000 },
    { "name": "批量B", "stationId": "ST-002", "type": "demand-response", "targetPowerKw": 2000 }
  ]
}
```

关键断言：

- `tree` 当前保护的是 `data.items`
- `simulate` 当前保护：
  - `estimatedRevenue`
  - `confidence`
  - `revenueRange`
  - `breakdown`
- `batch-simulate` 当前保护：
  - `totalEstimatedRevenue`
  - `items`

它在防什么：

- Strategy mock service 的结构回归

也要注意：

前端策略配置页期望的字段命名和这里保护的字段命名并不完全一样，因此当前仍存在契约漂移风险。

## 4. 推荐阅读顺序

如果是第一次接手项目，建议按这个顺序看测试：

1. `SystemHealthControllerTest`
2. `ProductionMonitorControllerTest`
3. `StationArchiveControllerTest`
4. `DashboardControllerTest`
5. `ForecastControllerTest`
6. `StrategyControllerTest`

原因：

- 前 3 个最能体现真实链路
- 后 3 个最能体现当前 mock 边界和契约风险

---

## 5. 当前测试体系的优点与缺口

### 优点

1. 每个模块至少有 controller 级自动化覆盖。
2. 真实 H2 链路不是空口说明，而是有集成测试保护。
3. 数组长度、关键字段名、固定示例值都有基本保护。

### 缺口

1. 还没有系统的异常分支测试。
2. 还没有权限、租户隔离测试。
3. 还没有系统化的前端浏览器截图回归。
4. `forecast`、`strategy` 的后端测试目前保护的是 mock 契约，不等于前后端页面联调已经稳定。
