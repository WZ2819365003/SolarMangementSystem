# 系统总览与学习路径

这份文档不是替代实现说明书，而是告诉一个第一次接手这个项目的人：

1. 这个系统到底分成哪些块。
2. 现在哪些块是真的，哪些块还是演示态。
3. 应该按什么顺序学习和维护。

## 1. 一句话理解这个系统

这是一个“光伏运营管理前后端同仓库项目”，目标是给虚拟电厂/资源聚合商提供：

- 综合监控
- 生产监控
- 电站监控
- 预测分析
- 策略管理
- 设备监测
- 告警中心

## 2. 当前项目分层

### 前端

- 技术栈：Vue 2、Vue Router 3、Vuex 3、Element UI、ECharts、高德地图 JS SDK
- 入口：`frontend/src/main.js`
- 路由：`frontend/src/router/index.js`
- API：`frontend/src/api/pvms.js`
- 运行桥接：`frontend/src/shared/host/bridge.js`

### 后端

- 技术栈：Spring Boot 3.2.5、Spring Web、Validation、JdbcClient、H2
- 入口：`backend/src/main/java/cn/techstar/pvms/backend/PvmsBackendApplication.java`
- 公共响应壳：`backend/src/main/java/cn/techstar/pvms/backend/common/ApiResponse.java`

### 数据

- 当前使用内嵌 H2 内存库
- 表定义：`backend/src/main/resources/schema.sql`
- 种子数据：`backend/src/main/resources/data.sql`

## 3. 当前模块成熟度

下表是现在最重要的真实状态：

| 模块 | 前端状态 | 后端状态 | 数据来源 | 备注 |
| --- | --- | --- | --- | --- |
| dashboard | 已接真实接口 | 地图相关部分 H2，其他部分 mock | H2 + mock | 同一模块内部混合 |
| production-monitor | 已接真实接口 | H2 + JDBC + 计算 | H2 | 当前最完整 |
| station-archive | 已接真实接口 | H2 + JDBC + 插值/汇总 | H2 | 当前最复杂 |
| forecast | 已有页面 | mock service | 内存生成 | 还没落库 |
| strategy | 已有页面 | mock service | 内存生成 + 内存状态 | 创建/提交后只保存在内存 |
| devices | 已有页面 | mock service | 内存列表 | 只读 |
| alarms | 已有页面 | mock service | 内存列表 | 只读 |
| resource-units | 已有页面接口 | mock service | 内存列表 | 列表/总览/曲线 |
| adjustable | 作为 dashboard 的能力补充 | mock service | 内存列表 | 未落表 |

## 4. 推荐学习顺序

### 第 1 步：先跑起来

看：

- `docs/01_部署与运行说明.md`

目标：

- 能自己启动本地前后端
- 知道日志在哪
- 知道如何停服务
- 知道 docker compose 的角色

### 第 2 步：先学最真实的模块

推荐顺序：

1. `production-monitor`
2. `station-archive`
3. `dashboard`

原因：

- 这三个模块最能代表当前系统的真实后端写法
- 它们覆盖了 controller -> service -> repository -> H2 -> SQL 种子数据 的完整链路

### 第 3 步：再学仍然是 mock 的模块

推荐顺序：

1. `forecast`
2. `strategy`
3. `devices` / `alarms`
4. `resource-units`
5. `adjustable`

学习重点不是“表怎么查”，而是：

- 它们当前返回的数据结构是什么
- 前端页面依赖哪些字段
- 以后如果要落库，应该补哪些表和服务

## 5. 新人接手时最容易误判的点

### 5.1 不要把“页面能打开”误认为“模块已完成”

这个项目里有不少页面已经做出来了，但后端还是 mock。  
所以“能看见页面”不代表“已有真实业务链路”。

### 5.2 不要把“后端有 controller”误认为“已经接数据库”

例如：

- `forecast`
- `strategy`
- `devicealarm`

这些模块虽然已经有 controller 和 test，但 controller 背后仍然是 mock service。

### 5.3 不要把“有数据库”误认为“数据可持久保存”

当前 H2 配置是内存库，不是文件库。  
后端一停，所有数据都会按 SQL 重新生成。

### 5.4 不要忽略“页面和接口之间的契约”

前端页面不是直接访问数据库，也不是直接访问 service。  
页面依赖的是 `frontend/src/api/pvms.js` 和每个接口的返回结构。  
以后改字段时，必须同时看：

- 前端页面用了什么字段
- 后端接口返回什么字段
- 测试断言了什么字段

## 6. 维护时的推荐动作顺序

如果你要改一个模块，建议按这个顺序走：

1. 先看本页对应的前端页面和后端接口。
2. 再看 `docs/02_后端实现说明书.md` 里的服务和仓储链路。
3. 再看 `docs/04_数据库设计说明书.md` 里的表字段定义。
4. 最后看 `docs/05_后端接口测试说明书.md`，确认已有测试怎么保护这条链。

## 7. 当前最值得继续补强的方向

### 7.1 把 mock 模块逐步落到表

优先级建议：

1. `forecast`
2. `strategy`
3. `devicealarm`
4. `resource-units`
5. `adjustable`

### 7.2 把 H2 内存库升级成可持久化环境

推荐方向：

- 开发环境继续保留 H2
- 联调/测试环境改为 MySQL 或 PostgreSQL
- 用 Flyway 或 Liquibase 管理版本化 SQL

### 7.3 把前端自动化补起来

当前仓库里已经有 Playwright 目录，但截图式、页面级回归还没有系统化写完整。  
后续建议把这些页面的关键态做成截图回归：

- 综合监控首页
- 生产监控各 tab
- 电站监控树和右侧详情
- 预测分析三页
- 策略管理三页
- 设备监测与告警中心

## 8. 一句话总结

这个项目已经不是纯 demo，但也还不是全量真实系统。  
最准确的理解方式是：

- 前端框架和页面骨架已经比较完整
- 后端已有一部分真实的 H2/JDBC 模块
- 另一部分模块仍然停留在 mock/演示态
- 现在最适合做“逐模块去黑盒化”和“逐模块落真链路”
