# 后端实现说明

## 1. 后端公共结构

### 基本技术结构

- Spring Boot 3.2.5
- Controller + Service + Repository 的轻量分层
- 数据访问以 `JdbcTemplate/H2` 为主
- 统一返回结构通过 `ApiResponse.success(...)` 输出：

```json
{
  "code": 0,
  "message": "success",
  "data": {}
}
```

### 当前数据库定位

- 现在的 H2 是开发联调数据底座。
- 一部分数据是“事实数据种子”，比如电站、逆变器、曲线、调度记录。
- 一部分数据是“业务元数据种子”，比如预测值、策略快照、收益快照。
- 服务层会基于这些表再做聚合、映射和衍生计算。

## M02 生产监控

### 代码入口

- Controller：
  - `backend/src/main/java/cn/techstar/pvms/backend/module/productionmonitor/controller/ProductionMonitorController.java`
- Service：
  - `ProductionMonitorDataService.java`
  - `ProductionMonitorMockService.java`

### 已实现接口

- `GET /api/pvms/production-monitor/meta`
- `GET /api/pvms/production-monitor/overview`
- `GET /api/pvms/production-monitor/output`
- `GET /api/pvms/production-monitor/load`
- `GET /api/pvms/production-monitor/dispatch`
- `GET /api/pvms/production-monitor/weather`

### 主要数据表

- `pm_resource_unit`
  字段重点：`id/name/region/city/status/dispatch_mode/strategy_label/latest_alarm_title/alarm_total`
- `pm_station`
  字段重点：`resource_unit_id/name/capacity_mw/weight/status/online_rate/alarm_count/load_visible/load_base_kw`
- `pm_station_realtime_snapshot`
  字段重点：`realtime_power_kw/load_kw/availability/health_grade`
- `pm_station_curve_15m`
  字段重点：`load_kw/pv_power_kw/forecast_power_kw/baseline_power_kw/irradiance/temperature`
- `pm_weather_snapshot`
  字段重点：`weather/cloudiness/temperature/irradiance/humidity/wind_speed/conclusion`
- `pm_dispatch_record`
  字段重点：`issued_at/command_type/target_power_mw/actual_power_mw/response_seconds/status/deviation_reason`

### 处理逻辑

- `meta` 负责给前端筛选条件、资源单元信息和默认值。
- `overview` 汇总资源单元级 KPI、告警和成员站概览。
- `output` 基于 15 分钟曲线聚合出实际出力、预测出力和站点贡献。
- `load` 会计算总负荷、总光伏出力、总可调空间、平均爬坡速率，并生成站点表和抽屉曲线。
- `dispatch` 汇总调度记录、执行成功率、响应时间和偏差原因。
- `weather` 输出天气摘要和天气趋势。

### 返回重点

- `overview`：Hero 区、状态条、成员站表、摘要卡
- `output`：功率曲线、成员站贡献
- `load`：KPI、站点行、弹窗曲线
- `dispatch`：指令记录、状态统计
- `weather`：天气摘要、辐照度 / 温度趋势

- `deviation-heatmap` 输出偏差热力图格子数据。
- `adjustable` 输出 24 小时可调能力和可调窗口。
- `accuracy` 输出精度趋势、误差分布、精度排名。

### 返回重点

- KPI
- 曲线序列
- 热力图矩阵
- 可调窗口

- 统计设备型号占比并生成运维建议。

- 生成 `processBoard` 作为处置建议面板。

### 返回重点

## 3. 后端维护时最重要的认知

- 现在后端不是“全真业务系统”，而是“H2 事实表 + 元数据 + 服务层计算”模式。
- 做新模块时，优先先定事实表、元数据表、响应契约，再写 Controller。
- 如果某个页面看起来像是“纯 mock”，先查这个模块有没有对应的 `*DataService`；有的话优先看 H2 链路，没有的话再看 `*MockService`。
