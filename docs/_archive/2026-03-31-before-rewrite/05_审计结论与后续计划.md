# 审计结论与后续计划

## 1. 当前结论

项目现在已经不是“纯前端 demo + 假接口”了。  
更准确的状态是：

- 前端模块完整度较高
- 后端已完成多模块 H2 化
- 开发环境默认联调
- 仍处在开发联调 / 预发布前阶段

## 2. 现状分类

### A. 已经完成前后端联调闭环的模块

- M02 生产监控
- M02 电站监控 / 电站档案
- M03 预测分析
- M04 策略管理
- M05 设备监测
- M06 告警中心

### B. 已完成部分 H2 化，但仍保留旧 mock 逻辑的模块

- M01 综合监控中心

原因：

- 地图点位、告警流、VPP 节点状态已经接 H2
- 但 dashboard 其他旧接口仍有部分走 `DashboardMockService`

### C. 仍偏 mock 的旧接口

- `resource-units` 相关接口仍主要靠 `StationMockService`

## 3. 之前那个桥接问题现在的真实状态

之前的风险是：

- 开发环境默认绕过真实后端，导致前后端契约漂移不容易暴露

当前代码状态已经变成：

- 默认走后端代理
- 宿主环境优先走宿主 `apirequest`
- 只有显式切到 `VUE_APP_REQUEST_MODE=mock` 才走前端 mock

所以这条风险现在不是“当前阻断问题”，更准确的说法应改成：

- 已修正默认行为
- 但团队仍需要约束：开发联调不要随手切回 mock 模式

## 4. 现在离 release 还差什么

### 工程层面

- 前端 lint 还没有完全收口
- 包体较大
- 文档现在已经收敛，但还需要团队按这套新文档持续维护

### 业务层面

- 真实数据库没有接入
- 外部系统取数链路没有接入
- 实时消息 / 任务 / 权限 / 审计日志没有完善
- 设备监测和告警中心目前还是只读，没有派单、确认、闭环动作

### 发布层面

- 缺少正式环境配置管理
- 缺少部署回滚方案
- 缺少系统级监控和告警
- 缺少 UAT 和业务验收脚本

## 5. 现在最值得优先做的事

### 第一优先级

- 把 dashboard 其余旧接口从 `DashboardMockService` 迁到 H2 / 真实服务层
- 把 `resource-units` 旧接口迁掉

### 第二优先级

- 给设备监测和告警中心补写操作接口
- 至少补：
  - 告警确认
  - 告警派单
  - 告警关闭
  - 设备处置记录

### 第三优先级

- 从 H2 迁移到正式数据库
- 把“事实数据种子”替换成外部系统取数
- 保留“元数据 + 服务层计算”的模式

## 6. 建议的长期边界

### 可以继续作为元数据维护的内容

- 预测模型定义
- 策略快照
- 收益模拟参数
- 展示用基准数据

### 更适合从外部系统实时 / 准实时获取的内容

- 光伏生产实时出力
- 设备在线状态
- 设备告警
- 调度执行记录
- 实际收益 / 实际结算

### 明确应由后端计算的内容

- KPI 汇总
- 精度、偏差、合格率
- 可调空间
- 收益模拟和收益对比
- 告警等级映射、处置建议、闭环率

## 7. 交接给新人的一句话建议

先把这套系统理解成“前端完整、后端在逐步去 mock 化的开发联调版”，不要一上来就按正式生产系统思路去找真实数据库和真实业务流。  
新同学上手时，先看 `01 -> 02 -> 03 -> 04` 这四份文档，再进代码，会快很多。
