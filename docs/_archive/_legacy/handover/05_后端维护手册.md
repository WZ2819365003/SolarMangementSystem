# 后端维护手册

## 1. 当前后端是什么状态

当前后端不是“纯壳子”，也不是“已经完成生产化”的业务系统。

更准确的判断是：

- 它是一个真实可运行的 Spring Boot 服务
- 不同模块成熟度不同
- 一部分模块仍然是 mock / 内存态实现
- 一部分关键链路已经进入 “H2 种子数据 + 后端计算” 阶段

截至 `2026-03-30`，至少下面三条链路已经不是前端 mock：

1. `dashboard` 地图动态点位链路
2. `production-monitor` 资源总览链路
3. `station-archive` 电站监控链路

## 3. 模块实现模式

当前后端主要有三种实现模式。

### 第一种：纯 mock / 演示装配

特征：
- controller 调用 `*MockService`
- 数据不落库
- 主要返回演示结构

适用：
- 快速起页面
- 演示型接口

风险：
- 很容易和前端字段漂移
- 本地联调不能验证真实契约

### 第二种：H2 种子数据 + 后端计算

特征：
- 基础事实数据落在 H2
- repository 负责读表
- service 负责 KPI、图表、摘要、风险提示计算

当前已落地：
- `dashboard`
- `production-monitor`
- `station-archive`

这类模块现在最适合继续演进，因为它们已经具备：
- 数据模型
- 本地联调能力
- 测试入口
- 替换真实数据源的边界

### 第三种：内存态业务服务

特征：
- 服务进程内维护状态
- 支持 create / submit / terminate 这类动作
- 重启后状态丢失

当前典型模块：
- `strategy`

## 5. 为什么要引入 H2

引入 H2 的目的不是做一个“临时数据库”，而是把下面几件事从前端 mock 拿回后端：

- 接口契约校验
- 本地真实联调
- 指标计算
- 数据模型沉淀
- 未来真实系统替换点

没有这层的话，页面虽然能展示，但无法验证：

- 查询参数有没有真的参与计算
- 字段结构是否稳定
- 哪些是元数据，哪些是计算结果

## 7. 当前最重要的限制

### 不是所有模块都已经数据化

不要因为 `dashboard`、`production-monitor`、`station-archive` 已经用了 H2，就误以为全仓都成熟到同一水平。

目前依然是多种实现模式并存。

### 仍然大量使用 `Map<String, Object>`

优点：
- 改得快
- 联调灵活

缺点：
- 字段名没有编译期约束
- 前后端契约漂移更难提前暴露

### 没有完整权限与审计

当前不要默认认为已经具备：
- 登录
- 权限控制
- 操作审计

这些都不在当前成熟范围内。

## 8. 当前最关键的扩展边界

### 适合落库的内容

- 公司 / 资源单元 / 电站 / 设备主数据
- 生产时序
- 预测时序
- 天气快照
- 调度记录
- 告警快照
- 策略快照

### 适合运行时计算的内容

- KPI 汇总
- 预测偏差
- 准确率
- 可调空间
- 风险提示
- 图表拼装
- 1 分钟插值

## 9. 新人第一次接手后端建议怎么做

1. 跑 `mvn test`
2. 打开 `http://127.0.0.1:8091/api/system/health`
3. 打开 H2 Console
4. 查看 `dashboard_*`、`pm_*`、`sa_*` 这三组表
5. 手动请求下面几个接口

- `/api/pvms/dashboard/stations-geo`
- `/api/pvms/production-monitor/load`
- `/api/pvms/station-tree`
- `/api/pvms/station-archive/station-realtime?stationId=SZ-001&metric=adjustable`
- `/api/pvms/station-archive/inverter-realtime?inverterId=SZ-001-INV-01`

## 10. 后续建议

近一步最值得做的不是继续堆 mock，而是：

1. 给 `production-monitor` 和 `station-archive` 补 DTO
2. 统一状态枚举和指标公式
3. 继续推进 `forecast`、`strategy` 从 mock 走向 H2 或真实 repository
