# 2026-03-29 前后端代码审计报告

## 审计范围

- 前端：`frontend/src`
- 后端：`backend/src/main/java`
- 测试与构建验证：
  - `backend`: `mvn test`
  - `frontend`: `npm run lint -- --no-fix`
  - `frontend`: `npm run build`

## 结论摘要

当前工程已经具备较完整的前端展示层和后端 Mock Controller 层，但“前后端已打通”这一点并不成立。核心问题不是单纯还在用 Mock，而是前端和后端已经分别演化出两套不同的数据契约；同时，开发态请求桥默认只走前端本地 Mock，导致这些分叉长期无法被联调暴露。

## 关键问题

### P1. M03 预测分析模块前后端契约已经分叉

前端 `ForecastOverviewView` 明确依赖 `viewData.kpis`、`viewData.stationTable`、`viewData.comparison`、`viewData.heatmap.data` 这套结构，见：

- `frontend/src/modules/forecast/components/views/ForecastOverviewView.vue:85`
- `frontend/src/modules/forecast/components/views/ForecastOverviewView.vue:99`
- `frontend/src/modules/forecast/components/views/ForecastOverviewView.vue:148`
- `frontend/src/modules/forecast/components/views/ForecastOverviewView.vue:259`

但后端 `ForecastMockService` 返回的是另一套结构，例如：

- `meta` 返回 `models`、`granularityOptions`，而不是前端在过滤栏中优先消费的区域信息，见 `backend/src/main/java/cn/techstar/pvms/backend/module/forecast/service/ForecastMockService.java:25`
- `overview` 返回 `kpi`、`accuracyTrend`、`forecastVsActual`，而不是前端视图使用的 `kpis`、`stationTable`、`comparison`，见 `backend/src/main/java/cn/techstar/pvms/backend/module/forecast/service/ForecastMockService.java:64`
- `adjustable` 返回 `summary` + `timeline`，而前端视图依赖 `kpis` + `capacityCurve` + `stationTable`，见 `backend/src/main/java/cn/techstar/pvms/backend/module/forecast/service/ForecastMockService.java:98`
- `deviation-heatmap` 返回 `matrix`，而前端消费字段是 `heatmap.data`，见 `backend/src/main/java/cn/techstar/pvms/backend/module/forecast/service/ForecastMockService.java:167`

影响：

- 当前页面之所以能工作，依赖的是前端本地 mock。
- 一旦切到宿主真实请求或本地 Spring Boot 联调，M03 页面会直接出现空表、空图或字段错位。

### P1. M04 策略管理模块前后端契约分叉更严重

前端 `StrategyConfigView` 和 `StrategyListView` 对接口结构有明确假设：

- 树接口返回 `res.data.tree`，节点字段是 `label`、`nodeType`、`statusTag`、`extra`，见 `frontend/src/modules/strategy/components/views/StrategyConfigView.vue:404`
- 电价接口中的每条记录要带 `period` 字段，见 `frontend/src/modules/strategy/components/views/StrategyConfigView.vue:503`
- 批量模拟请求体发送 `{ strategies: [...] }`，返回 `results` 和 `totalRevenue`，见 `frontend/src/modules/strategy/components/views/StrategyConfigView.vue:540`
- 单次模拟结果依赖 `confidenceRange` 和 `breakdown.peakSaving`，见 `frontend/src/modules/strategy/components/views/StrategyConfigView.vue:214`
- 批量创建请求体发送 `{ strategies: [...] }`，见 `frontend/src/modules/strategy/components/views/StrategyConfigView.vue:590`

后端 `StrategyMockService` 的契约则是：

- 树接口返回 `items`，节点字段是 `name` 和 `type`，见 `backend/src/main/java/cn/techstar/pvms/backend/module/strategy/service/StrategyMockService.java:101`
- 电价接口返回的 `periods` 元素只有 `start/end/type/typeLabel/price/action`，见 `backend/src/main/java/cn/techstar/pvms/backend/module/strategy/service/StrategyMockService.java:158`
- 批量创建和批量模拟都读取 `items`，不是 `strategies`，见 `backend/src/main/java/cn/techstar/pvms/backend/module/strategy/service/StrategyMockService.java:184`
- 单次模拟返回 `confidence` 和 `revenueRange`，不是 `confidenceRange`，见 `backend/src/main/java/cn/techstar/pvms/backend/module/strategy/service/StrategyMockService.java:243`
- 批量模拟返回 `totalEstimatedRevenue` 和 `items`，不是 `totalRevenue` 和 `results`，见 `backend/src/main/java/cn/techstar/pvms/backend/module/strategy/service/StrategyMockService.java:264`

影响：

- 策略树、批量配置、收益预览、批量模拟、批量创建这条链路在真实联调下会整体失效。
- 当前后端测试全绿，并不代表前端真实能消费这些接口。

### P1. 请求桥在开发态默认绕过后端，联调问题无法提前暴露

前端请求桥只有在“宿主提供 `window._tsx.apirequest` 且 `NODE_ENV=production`”时才走宿主请求；其他场景全部退回前端本地 `mockHandlers`：

- `frontend/src/shared/host/bridge.js:49`
- `frontend/src/shared/host/bridge.js:59`
- `frontend/src/shared/host/bridge.js:61`

影响：

- 本地开发、页面调试、Playwright 检查、前端 build 通过，都不能证明接口已和 Spring Boot 对齐。
- 契约分叉会持续累积，直到接入宿主或切换真实请求时集中爆炸。

## 优化建议

### 第一优先级

- 先统一 M03、M04 的接口契约，以前端或后端其中一侧为基准，另一侧一次性对齐。
- 给关键接口补上 DTO/VO，而不是继续用 `Map<String, Object>` 裸传。
- 给请求桥增加“开发态直连本地后端”的明确开关，避免开发阶段长期只打前端 mock。

### 第二优先级

- 把前端本地 mock 和后端 mock 收敛到单一数据源，至少做到字段定义共享。
- 补齐 `.gitignore`，排除 `backend/target/`、`frontend/tests/playwright/output/`、`frontend/tests/playwright/test-results/`。
- 先把 lint 的 3 个 error 清零，再处理高频 warning。

### 第三优先级

- 拆分超大页面和超大 mock 文件，尤其是 `StrategyConfigView.vue`、`ProductionMonitorPage.vue` 和几个大型 mock 数据文件。
- 结合构建结果做前端分包与依赖治理，优先看 `chunk-vendors` 体积。
- 对“前端已完成但后端未接入”的模块，建立一份真实联调清单，避免继续口径漂移。
