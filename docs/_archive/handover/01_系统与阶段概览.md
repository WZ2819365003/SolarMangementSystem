# 系统与阶段概览

## 1. 项目是什么

本项目是一个光伏管理系统，业务表达上更接近“虚拟电厂聚合商视角的光伏运营平台”，而不是单一电站监控页面。当前代码覆盖了以下几类能力：

- 综合监控中心
- 生产监控
- 预测与分析
- 策略管理
- 设备监控
- 告警中心

从仓库结构上看，它由两个主应用组成：

- `frontend/`：Vue 2 微前端子应用
- `backend/`：Spring Boot 3 后端服务

## 2. 当前处于什么阶段

如果按软件交付阶段来判断，这个项目更接近“Alpha 后期 / Beta 前的联调准备阶段”。

### 已经具备的东西

- 前端主页面、菜单、路由和模块入口基本齐全
- 后端接口分模块组织，Controller 和测试骨架较完整
- 后端已有 MockMvc 测试，`mvn test` 可以通过
- 前端可独立运行，也兼容被宿主系统挂载
- 设计文档、交付文档和审计文档已经沉淀到 `docs/`

### v2.0.0 新增

- 后端持久层已从 H2/JdbcClient 迁移至 **MyBatis + MySQL**
- 所有 MockService 和 Repository 已替换为 27 个 Mapper 接口
- 49 项 MockMvc 测试全部通过
- MySQL 种子数据脚本 `mysql-data.sql` 可用于生产初始化

### 还没有完成的关键闭环

- 权限、审计日志、外部系统接入层尚未接入
- M03 预测与分析、M04 策略管理仍存在部分前后端字段不一致
- Service 层大量使用 `Map<String, Object>`，缺乏编译期契约保护

## 3. 当前系统的真实运行形态

这个项目不是“纯前端假页面”，也不是“完整生产后端”。

更准确的定义是：

- 前端：真实页面 + 真实路由 + 真实请求封装 + 开发态 mock 数据
- 后端：真实 Spring Boot 接口 + 简化业务装配 + mock 数据源 / 内存态状态

因此它适合做：

- 需求评审
- 交互走查
- 功能演示
- 联调前准备
- 后续真实化改造

暂时不适合直接当作生产版本上线。

## 6. 新人接手时最重要的判断

接手这个项目时，先不要急着改页面或补接口。先确认下面三件事：

1. 你当前是在维护“演示态系统”，不是生产态系统。
2. 页面看到的数据，不代表后端契约就真的成立，因为前端开发态默认先走自身 mock。
3. 后续工作的优先级应该先放在契约收敛、联调闭环和数据源真实化，而不是继续堆页面。

## 7. 推荐的维护优先级

### 第一优先级

- 收敛 forecast 和 strategy 的前后端请求体、响应体
- 让前端开发环境能够稳定校验真实后端返回

### 第二优先级

- 梳理后端各模块从 `MockService` 到真实 Service/Repository 的演进路线
- 为关键接口补 DTO，而不是继续使用大面积 `Map<String, Object>`

### 第三优先级

- 收敛前端 lint 错误和告警
- 清理构建产物和测试产物管理
- 补充发布前的环境配置、日志、权限、安全边界
