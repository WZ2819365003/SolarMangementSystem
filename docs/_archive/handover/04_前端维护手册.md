# 前端维护手册

## 1. 前端是什么形态

前端不是普通的单页应用，而是“可独立运行，也可被宿主系统挂载”的微前端子应用。

关键入口：

- `frontend/src/main.js`
- `frontend/src/public-path.js`
- `frontend/src/shared/host/bridge.js`

`main.js` 里同时支持两种模式：

- 独立运行：直接在浏览器本地开发
- 宿主挂载：通过 qiankun / 宿主系统传入运行时能力

## 2. 页面骨架怎么工作

整体外壳组件是：

- `frontend/src/layouts/ShellLayout.vue`

它负责：

- 左侧菜单
- 顶部标题和面包屑
- 运行模式和当前用户信息展示
- `router-view` 内容承载

菜单并不是写死在布局里，而是来自：

- `frontend/src/settings.js`

因此如果你想：

- 改菜单名称
- 调整菜单顺序
- 新增一级入口

优先改 `settings.js`，不是直接改布局组件。

## 3. 路由层怎么维护

路由统一在：

- `frontend/src/router/index.js`

维护路由时要关注三个点：

1. `path`：页面地址
2. `component`：实际承载页面
3. `meta`：标题、所属 section、当前 viewKey、激活菜单等

本项目大量页面的子视图切换不是通过单独页面文件实现，而是：

- 路由切换 `viewKey`
- 页面组件根据 `viewKey` 动态加载具体视图组件

典型例子：

- `ForecastPage.vue`
- `StrategyPage.vue`
- `ProductionMonitorPage.vue`

## 4. 请求链路怎么走

前端请求链路统一是：

```text
页面组件 -> frontend/src/api/pvms.js -> frontend/src/shared/host/bridge.js -> 宿主 apirequest 或前端 mock
```

### `frontend/src/api/pvms.js`

这个文件相当于前端的接口总表：

- 所有 `/pvms/*` 请求基本都从这里定义
- 页面层不要到处散写请求 URL，尽量统一从这里走

### `frontend/src/shared/host/bridge.js`

这是前端维护时必须读懂的文件。

它做了三件事：

1. 判断当前是否运行在宿主环境
2. 构造运行时上下文（租户、用户名、主题等）
3. 决定请求到底走宿主真实接口，还是走前端 mock

当前最关键的事实是：

- 只有在 `production` 且宿主提供 `apirequest` 时，它才真正调用宿主请求
- 其余情况下会回退到前端 `shared/mock` 数据

这也是为什么本地开发时页面看起来“有数据”，但并不代表后端契约已经对齐。

## 5. 状态管理怎么维护

全局状态文件：

- `frontend/src/store/index.js`

当前 Vuex 使用得很轻，主要存：

- 应用名、版本、描述
- 菜单
- 运行时信息

也就是说：

- 页面业务数据大部分是组件内自己拉取
- Vuex 不是当前项目的复杂业务中心

如果后续业务状态越来越复杂，再决定是否把状态上收，不要提前过度抽象。

## 6. 前端模块的常见组织模式

当前前端模块大多采用类似结构：

```text
modules/<module>/
├─ pages/
├─ components/
└─ components/views/
```

典型职责划分：

- `pages/`：模块页面入口，负责拉取数据、切换子视图、组合组件
- `components/`：模块公共组件，如 Hero、TabNav、FilterBar
- `components/views/`：具体视图页签，如 overview/list/revenue

这套模式在以下模块最明显：

- `forecast`
- `strategy`
- `production-monitor`

## 7. 重点模块说明

### 综合监控中心

- 页面入口：`frontend/src/modules/dashboard/pages/OverviewPage.vue`
- 特点：地图、KPI、天气、告警和排名的聚合页

### 生产监控

- 页面入口：`frontend/src/modules/production-monitor/pages/ProductionMonitorPage.vue`
- 特点：一个页面同时承载“资源总览”和“电站监控”两种模式
- 难点：复用了 `stations` 模块中的树和右侧详情面板

### 预测与分析

- 页面入口：`frontend/src/modules/forecast/pages/ForecastPage.vue`
- 特点：overview / adjustable / accuracy 三个子视图
- 风险：与后端字段结构目前不一致

### 策略管理

- 页面入口：`frontend/src/modules/strategy/pages/StrategyPage.vue`
- 特点：list / config / revenue 三个子视图
- 风险：树结构、批量接口和模拟结果字段与后端不一致

## 8. 当前前端维护的主要风险

### 风险一：本地开发默认不校验真实后端

这会导致：

- 页面开发速度快
- 但接口真实度低
- 联调问题会延后暴露

### 风险二：M03 / M04 契约漂移

目前最明显的漂移点：

- Forecast：前端读 `kpis`、`stationTable`、`comparison.series`、`heatmap.data`
- Strategy：前端读 `tree`、`confidenceRange`、`results`、`totalRevenue`

这些字段与后端当前 mock 返回结构并不完全一致。

### 风险三：大文件较多

例如：

- `frontend/src/modules/strategy/components/views/StrategyConfigView.vue`
- `frontend/src/modules/production-monitor/pages/ProductionMonitorPage.vue`
- `frontend/src/shared/mock/*.js`

维护时容易出现：

- 改一个地方影响多个场景
- 阅读成本高
- 合并冲突频繁

## 9. 前端扩展时的建议做法

### 新增一个子模块

建议顺序：

1. 在 `settings.js` 决定是否暴露菜单
2. 在 `router/index.js` 补路由和 `meta`
3. 新建模块 `pages/` 入口
4. 在 `api/pvms.js` 增加统一接口方法
5. 明确该请求是走前端 mock、宿主接口，还是联调后端
6. 补至少一个 smoke 验证

### 修改已有接口

不要只改页面视图，必须同时检查：

- `api/pvms.js`
- `bridge.js`
- 对应 `shared/mock/*.js`
- 对应后端 Controller / Service
- 对应模块文档

## 10. 维护者最容易踩的坑

- 看到页面有数据，就误以为后端已经对齐
- 只改前端 mock，不改后端接口和文档
- 在 `router` 改了路径，但没有同步 `meta.activeMenu`
- 修改资源/电站页面时，没有意识到 `production-monitor` 也会复用相关组件
