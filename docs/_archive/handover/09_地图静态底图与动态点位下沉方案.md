# 地图静态底图与动态点位下沉方案

## 1. 这份文档解决什么问题

这个项目里的“地图”不是一个单独技术点，而是两类职责的组合：

- 前端负责地图资源和渲染。
- 后端负责地图上的业务数据。

如果不把这两个边界拆开，接手人很容易误以为“地图有问题就是前端问题”，或者继续把业务真相保存在前端 mock 里。这个专题文档就是用来明确边界、说明当前实现、解释后续扩展路径。

## 2. 当前边界

### 前端负责的静态层

前端继续负责：

- AMap SDK 加载和安全配置
- 底图样式
- 地图实例生命周期
- CircleMarker 绘制
- 用户点击、聚焦、重置视角
- AMap 资源失败时的 fallback 展示

对应代码入口：

- `frontend/src/shared/plugins/amap.js`
- `frontend/src/modules/dashboard/components/DashboardMapCard.vue`

### 后端负责的动态层

后端现在负责：

- 站点经纬度
- 站点所属资源单元
- 站点运行状态
- 实时功率
- 当日发电量
- 当日收益
- 地图筛选项
- 状态摘要统计
- 地图洞察区依赖的最近告警
- VPP 节点摘要

对应接口：

- `/api/pvms/dashboard/stations-geo`
- `/api/pvms/alarms/recent`
- `/api/pvms/dashboard/vpp-node-status`

## 3. 为什么这样拆

这样拆分有四个直接收益：

1. 前端不再保存地图业务真相。
2. 开发环境可以真实校验后端契约，不再被前端 mock 掩盖。
3. 将来切换数据库或外部系统时，不需要重写地图渲染层。
4. 排障时可以快速分清是“地图资源问题”还是“业务数据问题”。

## 4. 当前实现状态

dashboard 地图链路已经按这个边界落地：

- 前端底图仍然通过 AMap SDK 加载。
- 后端引入了内嵌 H2 数据库。
- dashboard 地图相关动态数据改为从 H2 读取。
- 一部分展示值由后端实时计算，而不是直接存表。

这说明当前开发环境已经具备“前端渲染 + 后端数据 + 后端计算”的完整链条，虽然仍然是开发种子数据。

- `today_energy_kwh`
- `today_revenue_cny`
- `availability`
- `health_grade`

### `dashboard_alarm_snapshot`

保存地图洞察区依赖的最近告警：

- `id`

- `adjustable_max_mw`
- `last_heartbeat`

## 6. 哪些值是存储事实，哪些值是后台计算

### 直接存储的事实

- 站点坐标
- 资源单元归属
- 站点额定容量
- 当前状态
- 当前实时功率
- 当前当日累计发电量
- 当前当日累计收益
- 告警记录
- VPP 节点基础快照

### 后端实时计算的值

- `filters.regionOptions`
- `filters.capacityOptions` 的筛选结果
- `summary[*].count`
- `statusLabel`
- `statusColor`
- `availableCapacityMw`
- `onlineStations`
- `totalStations`

这里要特别强调：真实生产环境下，这些统计和聚合值也应该继续由后端统一计算，而不是让前端重复写一套业务规则。

## 7. 为什么开发环境仍然要这样做

开发环境的数据虽然是造的，但逻辑链条不能是假链条。

这次的处理原则是：

- 原始业务事实可以在 H2 中造种子数据。
- 聚合指标和展示摘要要尽量在后端算。
- 前端只负责消费接口和渲染地图。

这样做的好处是，后面接入真实 GIS、SCADA、EMS、告警平台、预测平台、收益系统时，可以只替换数据来源，不用推翻接口边界。

## 8. 当前接口职责

### `/api/pvms/dashboard/stations-geo`

职责：

- 返回地图筛选项
- 返回站点状态摘要
- 返回地图点位列表

现在已经包含：

- `resourceUnitId`
- `resourceUnitName`
- `statusLabel`
- `statusColor`

### `/api/pvms/alarms/recent`

职责：

- 返回地图洞察区依赖的最近告警
- 返回告警级别汇总

### `/api/pvms/dashboard/vpp-node-status`

职责：

- 返回 VPP 节点基础信息
- 返回基于站点状态快照计算得到的可用容量和在线站点数

## 9. 排障顺序

地图问题优先按这个顺序排查：

1. AMap SDK 是否加载成功。
2. `/api/pvms/dashboard/stations-geo` 是否返回点位。
3. 站点点位是否带经纬度、状态、颜色相关字段。
4. `/api/pvms/alarms/recent` 是否返回地图洞察区数据。
5. `/api/pvms/dashboard/vpp-node-status` 是否返回节点摘要。

排障判断规则：

- 如果 AMap 失败但 fallback 还有点位，说明业务数据链路正常，问题在地图资源层。
- 如果 AMap 正常但没有点位，优先检查后端 H2、seed 数据和 service 计算逻辑。

## 10. 后续扩展路径

后面接真实系统时，建议按下面顺序替换：

1. `dashboard_station` 对接 GIS 或资产主数据。
2. `dashboard_station_status_snapshot` 对接 SCADA / EMS / 设备采集。
3. `dashboard_alarm_snapshot` 对接告警平台。
4. `dashboard_vpp_node_snapshot` 对接 VPP 主站或调度平台。

后续可新增但当前未落地的表：

- `weather_snapshot`
- `forecast_timeseries`
- `production_timeseries`
- `revenue_record`

这些表适合承接你提到的生产数据、设备数据、预测数据、收益数据等元数据或快照；但聚合指标、状态摘要、容量类结果，仍然建议继续保留在后端 service 中计算。

## 12. 一句话结论

现在的 dashboard 地图已经是“前端负责静态地图、后端负责动态点位和业务摘要”的结构。开发环境使用 H2 作为内嵌数据源，但关键指标和聚合逻辑已经下沉到后端，具备继续接真实系统的扩展基础。
