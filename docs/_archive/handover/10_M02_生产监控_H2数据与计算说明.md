# M02 生产监控 H2 数据与计算说明

## 1. 这篇文档解决什么问题

这篇文档专门讲 `production-monitor` 模块的三件事：

1. 现在后端的数据到底放在哪。
2. 哪些数据是开发态造的，哪些指标是在后端算的。
3. 以后如果接真实系统，应该替换哪一层，保留哪一层。

如果你是第一次接手这个项目，读完这篇文档之后，应该能够回答：

- 为什么负荷页现在不再 `404`
- 为什么说它已经不是纯 mock
- 为什么又不能说它已经接了真实业务系统

## 2. 当前总体架构

当前 `production-monitor` 的资源总览链路是：

```text
前端页面
  -> frontend/src/api/pvms.js
  -> frontend/src/shared/host/bridge.js
  -> /api/pvms/production-monitor/*
  -> ProductionMonitorController
  -> ProductionMonitorDataService
  -> Repository
  -> H2 内嵌数据库
  -> service 运行时聚合 / 计算 / 装配
  -> ApiResponse
  -> 前端视图组件
```

关键判断：

- 数据请求链路是真实后端链路。
- 基础数据放在 H2。
- 页面展示指标不是都直接来自表字段。

## 3. 为什么要这样做

如果继续沿用前端 mock，会有几个问题：

- 前端能看到页面，但无法验证真实接口字段。
- 后端即使少字段或字段错位，也不容易被发现。
- 一些本来应该在后端做的计算会被迫塞进前端。
- 后面接真实系统时，改造成本更高。

所以这次改造的目标不是“把数据简单搬到数据库”，而是明确分层：

- 元数据和原始事实数据放库里。
- 聚合指标和业务判断放后端 service。
- 页面只负责展示。

## 4. 当前已经落库的数据

`schema.sql` 里当前为 `production-monitor` 建了 6 张表。

### 4.1 `pm_resource_unit`

存资源单元级元数据：

- 单元 ID
- 单元名称
- 区域 / 城市
- 状态
- 聚合半径
- 调度模式
- 策略说明
- 最新告警摘要

用途：

- 筛选栏
- 资源单元头部信息
- 告警摘要
- 风险提示文案

### 4.2 `pm_station`

存站点主数据：

- 归属资源单元
- 装机容量
- 权重
- 状态
- 在线率
- 告警数
- 排序
- 是否展示在负荷页
- 负荷基础值

用途：

- 成员站列表
- 负荷页站表
- 可调能力计算
- 贡献排名

### 4.3 `pm_station_realtime_snapshot`

存某个时刻的站点实时快照：

- 快照时间
- 实时出力
- 实时负荷
- 可用率
- 健康等级

用途：

- Overview KPI
- Load summary
- 站点当前状态展示

### 4.4 `pm_weather_snapshot`

存资源单元天气快照：

- 天气
- 云量
- 温度
- 辐照度
- 湿度
- 风速
- 结论

用途：

- 天气卡片
- 天气趋势
- 风险和影响说明

### 4.5 `pm_station_curve_15m`

存 15 分钟级时序：

- 负荷
- 光伏出力
- 预测出力
- 基线出力
- 辐照度
- 温度

用途：

- Output 曲线
- Load 页电网交互曲线
- 当日累计电量
- 预测偏差率
- 最大爬坡速率

### 4.6 `pm_dispatch_record`

存调度执行记录：

- 下发时间
- 指令类型
- 目标功率
- 实际功率
- 响应时长
- 状态
- 偏差原因

用途：

- 调度执行 KPI
- 调度趋势
- 执行记录表

## 5. 当前后端实时计算的内容

这部分不是库里的现成字段，而是 `ProductionMonitorDataService` 计算出来的。

### 5.1 总览 KPI

包括：

- 实时总出力
- 当日累计电量
- 上调可调能力
- 下调可调能力
- 在线率
- 预测准确率

### 5.2 出力分析

包括：

- 粒度聚合后的曲线
- 预测偏差率
- 峰值出力
- 站点贡献排名

### 5.3 负荷与出力

包括：

- 总负荷
- 总光伏出力
- 总可调空间
- 平均爬坡速率
- 每个站点的电网交互曲线

### 5.4 调度执行

包括：

- 指令数
- 成功数
- 成功率
- 平均响应时长
- 今日可调上限
- 今日已调用量
- 风险提示

### 5.5 天气研判

包括：

- 天气趋势图
- 影响表
- 研判建议

## 6. 现在的数据哪些是“造的”

答案是：基础事实数据目前仍然是开发态造数。

这包括：

- 资源单元主数据
- 站点主数据
- 天气快照
- 15 分钟负荷 / 出力 / 预测曲线
- 调度记录

这些数据目前通过 `data.sql` 进入 H2。

所以：

- 现在不是接了真实外部系统。
- 但也不是前端自己编一份假数据。

更准确的说法是：

- 数据由后端在数据库中维护一份开发态事实集。

## 7. 现在后端为什么仍然有价值

因为真正重要的不是“数据是不是全真实”，而是：

- 计算放在哪一层
- 契约是否稳定
- 以后替换真实数据源时是否容易

当前后端的价值在于：

1. 它已经定义了页面真正依赖的接口结构。
2. 它已经承接了大部分指标计算。
3. 它已经把“原始数据”和“展示结果”分开了。

这意味着以后接真实系统时，你只需要替换数据源，不需要把整套逻辑再推翻一次。

## 8. 现在 `load` 页为什么这样设计

用户最早发现的问题是：

- `/api/pvms/production-monitor/load` 返回 `404`

现在这个接口已经补上，但它有一个当前阶段特性：

- 它接受 `resourceUnitId`
- 但还不按 `resourceUnitId` 过滤

原因不是漏写，而是现有前端 `load` 页更像一张“全局站表”，而不是“资源单元局部站表”。

所以当前策略是：

- 先保持前端展示完整
- 再逐步把交互收敛成真正按资源单元筛选

如果未来要改成按资源单元过滤，需要同时改：

1. 前端交互和文案。
2. 后端查询逻辑。
3. 对应测试。
4. 文档。

## 9. 粒度为什么在后端算

当前支持：

- `15m`
- `30m`
- `60m`

基础表只存 `15m` 数据。`30m` 和 `60m` 是 `ProductionMonitorSeriesAggregator` 在运行时聚合得到。

这样做的原因：

- 前端不需要自己拼粒度。
- 同一套规则可以复用在 output 和 load。
- 后面接真实历史库时，也更容易统一处理。

## 10. 扩展到真实系统时应该怎么替换

这是交接里最关键的延展性说明。

### 10.1 可以直接替换的数据源

这些表未来都可以由真实系统同步或采集填充：

- `pm_resource_unit`
- `pm_station`
- `pm_station_realtime_snapshot`
- `pm_weather_snapshot`
- `pm_station_curve_15m`
- `pm_dispatch_record`

替换来源可能包括：

- 资产台账系统
- 设备监控系统
- 生产采集系统
- 预测服务
- 天气服务
- 调度平台

### 10.2 应该继续保留在后端的逻辑

不要轻易改动以下职责归属：

- 粒度聚合
- KPI 汇总
- 预测偏差率
- 准确率
- 最大爬坡速率
- 风险提示
- 调度执行摘要

这些逻辑应该继续保留在后端 service。

### 10.3 后续适合新增的表

如果要继续往真实系统推进，建议考虑：

- 设备级实时表
- 设备级告警表
- 收益结算原始表
- 历史预测批次表
- 调度指令回执表

## 11. 维护时最容易踩坑的地方

### 误以为数据真实，所以不看 SQL

当前数据还是开发态种子数据。遇到数值异常时，不要第一时间怀疑前端，先看：

- `data.sql`
- H2 表数据

### 误以为现在还是纯 mock，所以不看 service

当前很多指标已经在 `ProductionMonitorDataService` 里算了。只改种子数据，不看 service，很容易误判结果。

### 误以为 `load` 的 `resourceUnitId` 不生效就是 bug

当前它确实没参与过滤，这是已知设计边界。

### 误以为 station 模式也已经接到同一套后端

没有。`/production-monitor/station` 还是 `stationarchive`。

## 12. 推荐的维护顺序

如果你下一步继续做 `production-monitor`，建议按下面顺序：

1. 先把前端 `load` 页的查询条件真正落到 URL 和交互上。
2. 再决定是否按 `resourceUnitId` 过滤负荷页。
3. 然后给 `overview/output/dispatch` 补更细的 DTO 和契约约束。
4. 最后再考虑接真实生产、预测、收益、设备数据源。

## 13. 一句话总结

`production-monitor` 现在已经从“前端 mock 页面”推进到了“后端持有开发态事实数据，并在后台完成主要指标计算”的阶段。后续真正要做的，不是再把计算搬回前端，而是逐步把 H2 中的开发态数据替换成真实业务数据源。
