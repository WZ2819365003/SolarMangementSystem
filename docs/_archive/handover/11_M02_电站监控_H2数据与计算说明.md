# M02 电站监控 H2 数据与计算说明

## 1. 这份文档解决什么问题

这份文档专门说明 M02 里的“电站监控模式”现在是怎么跑起来的。

目标不是讲页面长什么样，而是让接手的人快速回答下面 4 个问题：

1. 电站监控现在是不是还靠前端 mock
2. 后端哪些数据是直接造的
3. 哪些指标是后端运行时计算出来的
4. 未来要接真实系统时，从哪里替换

截至 `2026-03-30`，答案已经很明确：

- 不是纯前端 mock
- 开发环境底座是 H2
- 原始事实数据落库
- 图表和摘要指标由后端计算

## 2. 当前链路范围

这次落地覆盖的是 M02 的“电站监控模式”，不包含资源总览模式。

## 4. H2 表说明

### `sa_company`

保存公司主数据：
- 公司 ID
- 公司名称
- 区域
- 排序

### `sa_station`

保存电站主数据：
- 电站 ID
- 公司归属
- 电站名称
- 装机容量
- 电站状态
- 数据质量状态
- 并网状态
- 投运日期
- 地址
- 基础负荷
- 排序

### `sa_station_curve_15m`

保存 15 分钟粒度原始曲线：
- 负荷
- 实际光伏出力
- 日前预测
- 超短期预测

这是当前最重要的“原始事实数据表”。

### `sa_inverter`

保存逆变器主数据：
- 归属电站
- 额定功率
- 设备状态
- 型号 / 厂商 / 序列号
- MPPT
- 电压频率参数
- 组串数量
- 每串面板数

### `sa_station_strategy`

保存电站当前策略快照：
- 策略名称
- 策略类型
- 策略状态
- 开始/结束时间
- 目标功率
- 预计收益

### `sa_inverter_alarm`

保存逆变器告警快照。

## 5. 哪些是元数据，哪些是后端计算

### 直接造到数据库里的元数据

- 公司
- 电站
- 逆变器
- 当前策略快照
- 当前告警快照
- 15 分钟基础曲线

### 后端实时计算的数据

- 树节点上的实时负荷、实时光伏出力、可调空间
- 公司级 KPI
- 电站级 `adjustable` 视图的 KPI
- 月度统计
- VPP 可调字段
- 负荷分项
- 逆变器功率曲线
- 逆变器拓扑
- 策略执行日志
- 1 分钟粒度图表

### 为什么要这样拆

因为真实系统上线后，变化最大的通常是数据来源，不是页面结构。

将来如果接入：
- 资产系统
- 设备台账系统
- SCADA / EMS
- 预测服务
- 告警中心
- 调度平台

只需要替换 repository 或数据装配来源，不必推翻前端页面契约。

## 6. 当前计算逻辑说明

### 树节点当前值

树节点的“当前值”不是额外存表，而是从 `sa_station_curve_15m` 取 `14:00` 这一时刻：

- `loadKw`
- `pvOutputKw`
- `adjustableKw = loadKw - pvOutputKw`

### 电站 `adjustable` 视图

后端会基于 15 分钟曲线生成：

- `series`
- `stationKpis`
- `monthlyStats`
- `fields`

这里的 `monthlyStats` 和 `fields` 目前是“基于曲线和站点元数据推导”的开发环境计算值，不是外部系统回传值。

### 1 分钟粒度

数据库只存 15 分钟曲线。

当请求 `granularity=1min` 时，后端使用线性插值生成 1440 个点。

这层逻辑在：
- `StationArchiveSeriesService`

### 逆变器功率

逆变器功率曲线不是单独落库，而是：

1. 取电站总出力曲线
2. 按逆变器额定功率和设备状态分摊
3. 生成每台逆变器的曲线

这意味着当前逆变器曲线是“推导曲线”，不是采集曲线。

## 7. 现在和真实生产系统的差距

### 已经具备的能力

- 开发环境真实接口联调
- 数据模型落库
- 后端按契约装配
- 粒度转换
- 典型 KPI 计算
- 树 / 图表 / 详情页后端统一出数

### 还不具备的能力

- 实时采集接入
- 真实分钟级时序
- 历史归档
- 策略执行真实日志
- 真实收益结算
- 权限、审计、租户隔离

## 8. 后续接真实系统时怎么扩展

### 第一阶段

先替换原始事实来源：

- `sa_station` -> 资产主数据
- `sa_inverter` -> 设备主数据
- `sa_station_curve_15m` -> 采集/预测服务
- `sa_station_strategy` -> 调度平台当前策略
- `sa_inverter_alarm` -> 告警中心

### 第二阶段

再替换推导逻辑：

- 1 分钟插值 -> 真实分钟级曲线
- 逆变器分摊曲线 -> 真实逆变器采集
- 策略执行日志 -> 平台回执记录

### 第三阶段

再做工程化增强：

- DTO 化
- 枚举统一
- 异常处理统一
- 配置化参数
- 权限与审计

## 9. 当前验证方式

这次改造至少要验证下面几件事：

1. `mvn -Dtest=StationArchiveControllerTest test`
2. `station-tree` 能返回三层树
3. `station-realtime?metric=forecast&granularity=1min` 能返回 1440 点
4. `inverter-realtime` 只传 `inverterId` 也能成功

## 10. 新人接手建议

建议按这个顺序理解这条链：

1. 先读 `docs/modules/frontend/stations.md`
2. 再读 `docs/modules/backend/station-archive.md`
3. 打开 `schema.sql` 和 `data.sql`
4. 看 `StationArchiveDataService`
5. 最后跑 `StationArchiveControllerTest`

如果这 5 步能走通，就已经具备继续维护电站监控链路的基础能力。
