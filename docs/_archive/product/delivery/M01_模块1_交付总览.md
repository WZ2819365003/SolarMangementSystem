# M01 模块 1 交付总览

## 1. 模块范围

模块 1 对应首页驾驶舱，即 `综合监控中心`。本轮同时完成了：

- 前端骨架与宿主适配
- 首页 GIS 驾驶舱页面
- 首页所需的 Spring Boot mock 接口
- 前端 Playwright 布局校验

- GIS 电站总览
- 地图筛选
- 地图点位选中与局部放大
- 返回全景

- 调度摘要
- 区域态势
- 核心 KPI 看板
- 天气卡片
- 计划 vs 实际功率曲线
- 发电排名
- 实时告警与告警抽屉

## 2. 当前交付状态

### 2.1 前端

关键页面与路由：

注意：

### 2.2 后端

后端已经完成模块 1 对应的 mock API：

- `/api/system/health`
- `/api/pvms/dashboard/stations-geo`
- `/api/pvms/dashboard/kpi-summary`
- `/api/pvms/dashboard/power-curve`
- `/api/pvms/dashboard/station-ranking`
- `/api/pvms/alarms/recent`
- `/api/pvms/weather/current`

这些接口已经有 Spring Boot 测试覆盖。

### 2.3 一个必须说明清楚的现状

虽然模块 1 的后端接口已经写好，但当前前端在本地开发态 **默认不直连 Spring Boot**。

前端请求链路当前是：

- 生产态：优先走宿主注入的 `window._tsx.apirequest`
- 本地开发态：走 `frontend/src/shared/mock/data.js`

也就是说：

- 现在前端页面展示的数据，默认来自前端 mock
- 后端接口是作为“契约实现 + 测试基线”存在
- 如果后续要切到本地 Spring Boot 联调，需要调整 `frontend/src/shared/host/bridge.js`

这点对后续维护非常重要。

## 6. 后续模块建议继续沿用的模式

后续模块建议统一按下面的流程做：

1. 先在 `doc/` 下写设计稿
2. 前端先做页面和 mock，确保视觉和交互成立
3. 后端按接口契约补 Spring Boot controller + service
4. 后端每个功能都先写测试，再落实现
5. 前端每个页面都补 Playwright 布局校验
6. 模块完成后，把最终维护说明写到 `doc/finished/`

这样后续每个模块都能保留清晰的交付记录。
