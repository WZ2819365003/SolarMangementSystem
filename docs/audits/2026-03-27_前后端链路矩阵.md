# 光伏管理系统前后端链路矩阵

## 1. 说明

这份矩阵只回答一件事:

- 某个页面功能现在到底是
  - `前后端已打通`
  - `前端展示 + 后端部分支持`
  - 还是 `纯前端 mock`

## 3. 前端请求入口

统一入口:

- `frontend/src/api/pvms.js`

统一桥接:

- `frontend/src/shared/host/bridge.js`

当前行为:

1. 生产环境且宿主存在 `window._tsx.apirequest` 时，走宿主请求
2. 否则直接走 `frontend/src/shared/mock/data.js`

所以开发态默认判断标准不是“本地后端是否存在”，而是“mock handler 是否存在”。

## 5. 难度评估矩阵

| 功能缺口 | 难度 | 原因 |
| --- | --- | --- |
| 补 `production-monitor/load` 后端 | 中 | 单视图补接口，数据结构已明确 |
| 补 `devices/monitor` 后端 | 中 | 设备列表与统计较常规 |
| 补 `alarms/center` 后端 | 中 | 告警列表与筛选常规，但闭环处理会变复杂 |
| 补 `station-tree + station-archive` 整条链路 | 中高 | 涉及树结构、多对象、多图表、多时序 |
| 把模块 2 全量切成真实联调 | 高 | 需要先统一资源单元/电站/设备三层模型 |
| 接真实天气采集 | 中高 | 需要外部天气源、映射规则、时间粒度处理 |
| 去掉前端默认 mock 依赖 | 高 | 会一次性暴露大量未补齐接口 |
| 建数据库与真实业务后端 | 高 | 当前后端仅为 mock 契约层 |

## 6. 当前最值得优先补的链路

如果目标是“最快把工程从演示版拉向联调版”，建议优先做这 4 条:

1. `production-monitor/load`
2. `devices/monitor`
3. `alarms/center`
4. `station-tree + company-overview`

原因:

- 页面已经有
- 用户可感知强
- 补完后前后端打通度会明显提升
- 不需要先碰最重的数据库和权限体系
