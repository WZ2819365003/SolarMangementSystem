# 后端维护手册

## 1. 后端当前处于什么状态

当前后端不是“纯空壳”，也不是“已经完成生产化”的业务系统。

更准确的判断是：

- 它是一个真实可运行的 Spring Boot 服务。
- 接口层已经按业务模块分包。
- 一部分模块仍是 mock/内存态服务。
- 一部分模块已经开始进入“开发态真实链路”。

截至 `2026-03-30`，至少有两条链路已经不再是简单前端 mock：

1. `dashboard` 地图动态点位链路
2. `production-monitor` 资源总览链路

这两条链路都已经引入了 H2 内嵌数据库和后端装配逻辑。

## 2. 启动入口和基础配置

## 3. 统一响应体

统一响应封装：

- `backend/src/main/java/cn/techstar/pvms/backend/common/ApiResponse.java`

当前结构：

```json
{
  "code": 0,
  "message": "success",
  "data": {}
}
```

维护原则：

- 现阶段优先保持前端契约稳定。
- 如果后续要引入错误码、异常映射、分页结构，应在 `ApiResponse` 和全局异常层统一处理，不要散落到各个 controller。

## 4. 模块组织方式


- `stations`

- `system`

## 5. 现在后端其实有三种实现模式

这是交接时最重要的判断维度。

### 第一种：纯 mock / 演示装配

典型特征：

- controller 直接调用 `*MockService`
- service 内部拼 `Map<String, Object>`
- 数据不落库
- 适合页面演示，但不适合真实联调

当前仍大量存在于：

- `forecast`
- 部分 `dashboard`
- 部分 `stations`

### 第二种：H2 种子数据 + 后端计算

典型特征：

- 基础事实数据落在 H2
- repository 从表中取数据
- service 在运行时做聚合、统计、风险装配
- 适合开发环境联调和后续真实数据源替换

当前已经落地于：

- `dashboard` 地图动态点位链路
- `productionmonitor` 资源总览链路

### 第三种：内存态状态服务

典型特征：

- 服务进程内维护状态
- 支持 create / submit / terminate 这类动作
- 进程重启后状态丢失

当前典型模块：

- `strategy`

## 6. 为什么这次开始引入 H2

原因不是为了“做个临时数据库”，而是为了把以下几件事从前端 mock 中拿回来：

- 接口契约校验
- 真实请求路径联调
- 后端指标计算
- 数据模型演进
- 未来接外部系统时的替换点

如果没有这一层，前端页面即使能展示，也无法验证：

- 后端字段是否齐全
- 查询参数是否真正参与计算
- 数据链路里哪些是元数据，哪些是实时计算结果

## 7. `production-monitor` 的真实定位

`productionmonitor` 模块当前已经从“mock 契约层”进入到“开发态业务后端”。

它现在具备：

- H2 主数据和时序数据
- 负荷、出力、天气、调度记录
- 后端粒度聚合
- KPI 计算
- 风险提示装配

但它还没有具备：

- 真实历史库
- 实时采集链路
- 调度平台对接
- 收益结算链路
- 权限和审计

换句话说：

- 数据源还是造的
- 但计算逻辑已经在后端

这正是开发态应该有的边界。

## 9. 当前最重要的限制

### 不是所有模块都已经上数据库

不要因为 `dashboard` 和 `productionmonitor` 用了 H2，就误以为整个后端已经数据化。现在仍然是多种实现模式并存。

### 仍然大量使用 `Map<String, Object>`

这让开发很快，但风险也明显：

- 字段名容易漂移
- 编译期没有 DTO 约束
- 前后端不一致时，问题容易拖到联调时才暴露

### 没有认证鉴权

当前依赖里没有完整的安全体系。不要假设：

- 用户登录
- 权限校验
- 操作审计

已经落地。

### H2 只是开发态数据底座

H2 解决的是：

- 开发环境联调
- 数据表建模
- 后端计算承载

它不等价于生产数据库方案。

## 10. 如何新增或修改一个后端接口

建议按下面顺序做：

1. 先确认这个接口属于哪种实现模式：
   - 纯 mock
   - H2 + 计算
   - 内存态状态
2. 再确定应该放在哪个模块。
3. 如果是 H2 链路，先想清楚“哪些数据应该落库，哪些指标应该实时算”。
4. 修改 controller 之前，先补或改测试。
5. 修改 service/repository。
6. 跑模块测试和全量后端测试。
7. 更新模块文档和交接文档。

### 一个实用判断标准

以下内容更适合落库：

- 站点主数据
- 设备主数据
- 生产时序
- 预测时序
- 天气快照
- 调度记录
- 收益原始记录

以下内容更适合在后端运行时计算：

- KPI 汇总
- 偏差率
- 准确率
- 爬坡速率
- 在线率
- 可调能力
- 风险等级

## 11. 当前高风险模块

### `forecast`

- 已有 controller 和测试
- 但前后端契约尚未完全收敛
- 目前仍更偏 mock 输出

### `strategy`

- 功能动作较多
- 有内存态业务流
- 但字段契约和前端还没完全对齐
- 未持久化

### `stationarchive`

- 生产监控的“电站监控模式”仍依赖它
- 如果后续继续做生产监控深度联调，这里会是下一个重点

## 12. 推荐的演进方向

### 近期

- 把 `forecast` 和 `strategy` 的接口契约先收敛
- 继续沿用 “H2 种子数据 + 后端计算” 方式推进重要模块
- 为 H2 链路补 DTO 和更细的测试

### 中期

- 按模块逐步引入真实 repository/service 分层
- 把算法参数、环境配置从硬编码抽出来
- 引入统一异常处理和结构化日志

### 长期

- 接入真实外部系统：
  - 生产数据
  - 设备数据
  - 预测服务
  - 收益系统
  - 调度平台
- 引入权限、审计和发布级监控

## 13. 新人第一次接手后端的推荐动作

1. 跑通 `mvn test`。
2. 打开 `http://127.0.0.1:8091/api/system/health`。
3. 用 H2 Console 看 `dashboard` 和 `productionmonitor` 的表。
4. 手动请求一遍：
   - `/api/pvms/dashboard/stations-geo`
   - `/api/pvms/production-monitor/load`
5. 读完对应模块文档：
   - `docs/modules/backend/dashboard.md`
   - `docs/modules/backend/production-monitor.md`
6. 再看 `forecast` 和 `strategy` 的审计问题，不要一上来就以为全仓都已经是同一成熟度。
