# 后端维护手册

## 1. 后端的真实定位

当前后端不是“空壳”，但也不是完整生产后端。

更准确地说，它是：

- 真实可运行的 Spring Boot 服务
- 按模块组织的接口层
- 带有一部分演示态业务装配
- 以 mock 数据源和内存态状态为主

这意味着你可以：

- 跑通接口
- 做前后端联调
- 继续往真实服务演进

但不能假设它已经具备：

- 真实持久化
- 权限认证
- 审计日志
- 外部系统接入

## 2. 启动入口和基础配置

启动类：

- `backend/src/main/java/cn/techstar/pvms/backend/PvmsBackendApplication.java`

配置文件：

- `backend/src/main/resources/application.yml`

当前已知基础配置：

- 服务端口：`8091`
- 应用名：`pvms-backend`

## 3. 统一响应体

统一响应封装：

- `backend/src/main/java/cn/techstar/pvms/backend/common/ApiResponse.java`

当前结构非常简单：

```json
{
  "code": 0,
  "message": "success",
  "data": {}
}
```

维护建议：

- 如果后续要补错误码、异常映射、分页结构，可从这里开始统一扩展

## 4. 模块组织方式

后端当前按业务模块分包，常见结构是：

```text
module/<module-name>/
├─ controller/
└─ service/
```

已存在模块：

- `adjustable`
- `dashboard`
- `devicealarm`
- `forecast`
- `productionmonitor`
- `stationarchive`
- `stations`
- `strategy`
- `system`

## 5. 当前后端的开发模式

大多数组件路径是：

```text
Controller -> MockService -> 返回 Map<String, Object> -> ApiResponse.success(...)
```

这说明当前后端强调的是：

- 快速提供接口形态
- 支撑页面演示
- 支撑单元/接口测试

而不是：

- 强类型 DTO
- 分层实体模型
- 真实数据库查询
- 复杂领域规则

## 6. 目前的“业务逻辑”到底有多少

不能说完全没有业务逻辑，但逻辑主要分两种：

### 第一种：演示型生成逻辑

典型模块：

- `forecast`
- `dashboard`

常见方式：

- 使用固定站点列表
- 通过数学函数生成曲线
- 用 `Math.random()` 制造波动
- 拼装出图表和 KPI 所需数据结构

### 第二种：内存态状态逻辑

典型模块：

- `strategy`

这里不只是静态返回，还会：

- 初始化一批策略记录
- 允许 create / submit / terminate / batch 操作
- 在进程内维护策略状态

但注意：

- 这些状态不持久化
- 服务重启后会回到初始值

## 7. 后端当前最重要的限制

### 没有真实持久层

当前没有明显看到：

- `Repository`
- `JpaRepository`
- `JdbcTemplate`
- MyBatis Mapper

所以不要默认后端已经连接了数据库。

### 没有认证鉴权层

`pom.xml` 当前主要是：

- `spring-boot-starter-web`
- `spring-boot-starter-validation`
- `spring-boot-starter-test`

没有看到安全框架依赖，因此不要假设权限控制已经实现。

### 广泛使用 `Map<String, Object>`

这在 demo 阶段很灵活，但后续会带来：

- 字段难以追踪
- 编译期约束不足
- 前后端契约漂移不易发现

## 8. 如何新增或修改一个接口

建议按下面顺序做：

1. 明确接口属于哪个模块
2. 在对应 `controller/` 新增或修改映射
3. 在对应 `service/` 增加返回结构或业务逻辑
4. 优先为请求体和响应体设计 DTO
5. 在对应 `*ControllerTest.java` 补测试
6. 同步更新 `docs/modules/backend/` 和交接文档

## 9. 当前重点风险模块

### `forecast`

- 接口齐全
- 但返回结构和前端期望不一致

### `strategy`

- 接口和测试最丰富
- 但前端批量配置、树结构、收益模拟字段与后端不完全一致

## 10. 后端维护建议

### 近期建议

- 先补 DTO 和契约说明
- 先让 forecast / strategy 联调闭环
- 先把 mock service 的字段命名稳定下来

### 中期建议

- 引入真实 Service + Repository 分层
- 引入配置分环境管理
- 引入统一异常处理和日志

### 长期建议

- 接入真实数据源
- 引入认证鉴权
- 引入审计日志和操作追踪
