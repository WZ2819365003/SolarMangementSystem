## 4. `production-monitor` 现在要怎么排障

### 第一步：先确认是不是后端真的接住了

直接访问：

- `http://127.0.0.1:8091/api/pvms/production-monitor/meta`
- `http://127.0.0.1:8091/api/pvms/production-monitor/load?date=2026-03-30&granularity=15m`

如果后端地址正常，但前端页面没数据，再去看代理和页面消费。

### 第二步：确认前端是否走了开发代理

检查：

- `frontend/src/shared/host/bridge.js`
- `frontend/vue.config.js`

如果前端又退回 `shared/mock`，页面看起来可能“有数据”，但那不是当前后端链路。

### 第三步：确认 H2 是否真的有种子数据

打开：

- `http://127.0.0.1:8091/h2-console`

重点查看：

- `pm_resource_unit`
- `pm_station`
- `pm_station_realtime_snapshot`
- `pm_weather_snapshot`
- `pm_station_curve_15m`
- `pm_dispatch_record`

如果这些表为空，优先检查：

- `application.yml` 中的 `spring.sql.init.mode`
- `schema.sql`
- `data.sql`

### 第四步：确认是不是查询条件和当前实现不一致

当前 `load` 接口虽然接受：

- `resourceUnitId`
- `date`
- `granularity`

但只真正使用：

- `date`
- `granularity`

所以“传了 `resourceUnitId` 却没按资源单元过滤”是当前设计，不是 bug。

### 第五步：确认是不是前端抽屉字段结构坏了

`ProductionLoadView.vue` 的“查看”抽屉依赖：

- `gridInteraction.times`
- `gridInteraction.series`

如果表格能出来，但点击查看没有图，优先检查这里。

## 5. 常见问题定位

### 访问 `/api/pvms/production-monitor/load` 返回 `404`

当前正常情况下不应该再出现。如果出现，优先看：

1. 后端是否起在 `8091`
2. `ProductionMonitorController` 是否编译并生效
3. 前端是否请求到了错误端口或错误路径

### 页面有数据，但和后端返回不一致

大概率是：

- 前端没有真正走后端
- 或浏览器缓存了旧 bundle

先清：

- 浏览器缓存
- 重新启动前端 dev server

### H2 Console 打不开

先确认：

- 后端已启动
- `application.yml` 中 `spring.h2.console.enabled=true`
- 访问的是 `http://127.0.0.1:8091/h2-console`

### 查询某天负荷页没有数据

当前 H2 只内置了 `2026-03-30` 的时序数据。其他日期如果没有补种子数据，返回为空是预期行为。

### 为什么页面刷新后粒度丢了

因为前端当前没有把 `date` 和 `granularity` 写回 URL query。

## 6. 交接给下一位维护者前必须补齐什么

### 文档层面

- 更新对应模块文档
- 更新交接手册
- 如果改了表结构，必须写明：
  - 哪些是基础事实表
  - 哪些指标是后端实时算的
  - 哪些数据仍是开发态造数

### 验证层面

- 写明跑过哪些命令
- 写明访问过哪些接口
- 如果没跑前端或 Playwright，要明确说明原因

### 代码层面

- 不要只改前端不确认后端契约
- 不要只补数据库表不补 service 计算说明
- 不要把应由后端承担的聚合逻辑偷偷搬回前端

## 9. 对维护者最有用的一句判断

如果你碰到一个页面“能显示”，先不要默认它已经联通真实后端。先确认：

1. 前端是不是走了 `/api` 代理
2. 后端是不是 controller 真实现
3. 数据是不是来自 H2/后端计算，还是仍来自 mock

这个判断能避免你在错误层面上花很多时间。
