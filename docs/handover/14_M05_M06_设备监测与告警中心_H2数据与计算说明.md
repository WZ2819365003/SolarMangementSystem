# M05 M06 设备监测与告警中心 H2 数据与计算说明

## 1. 本次改造做了什么

这次把剩余两个模块收到了和前面模块一致的模式里：

- 前端页面继续保留现有结构
- 后端不再依赖前端 mock
- 数据底座复用现有 `station-archive` 的 H2 事实表
- 展示指标、分组比例、提示文案和标签归一化放在后端计算

## 2. 数据从哪里来

设备与告警没有新建一套孤立表，直接复用以下 H2 表：

- `sa_station`
- `sa_inverter`
- `sa_inverter_alarm`
- `sa_station_curve_15m`
- `sa_station_strategy`

这样做的好处是：

- 设备、站点、策略、告警处在同一条数据链上
- 后续替换成真实采集系统时，只需要替换 repository 数据来源
- 不会再出现前端页面一套数据、后端接口另一套数据的漂移

## 3. 后端实时计算了哪些东西

### 设备监测

后端实时计算：

- 设备总数、在线数、离线数、告警数
- 设备分组占比
- 运维提示
- 设备负荷率
- 展示温度
- 最近上报时间

说明：

- 负荷率不是种子表直接存的，而是按当前站点出力、逆变器数量、额定功率和设备状态因子计算
- 最近上报时间也是后端生成的稳定展示值，用于开发环境下保持页面可读

### 告警中心

后端实时计算：

- 今日新增
- 待处理数量
- 处理中数量
- 闭环率
- 处置看板文案
- 告警类别
- 前端使用的中文级别与状态
- 责任人标签

说明：

- H2 原始告警字段和前端页面选项并不完全一致
- 这层归一化必须留在后端，不能重新放回前端拼装

## 4. 当前接口

### 设备监测

- `GET /api/pvms/devices/monitor`

参数：

- `keyword?`
- `status?`：`在线` / `离线` / `告警`

### 告警中心

- `GET /api/pvms/alarms/center`

参数：

- `keyword?`
- `level?`：`严重` / `重要` / `一般` / `提示`
- `status?`：`未处理` / `处理中` / `待确认` / `已关闭`

## 5. 后续接真实系统时怎么扩展

如果后面接真实设备平台、告警平台，推荐这样替换：

1. 保留 controller 返回结构不变
2. 保留 service 里的聚合与计算逻辑
3. 只替换 repository 的数据源

优先替换顺序：

1. `sa_inverter` 的设备清单和实时状态来源
2. `sa_inverter_alarm` 的告警事实来源
3. `sa_station_curve_15m` 的站点出力来源
4. `sa_station_strategy` 的策略执行关联来源

## 6. 交接时要特别说明的限制

- 当前仍然是开发环境造数，不是生产实时采集
- 一部分展示字段是后端稳定生成值，不是实时硬件采样原值
- 当前页面是读接口，不包含工单派发、确认、关闭动作
- 如果未来要做闭环操作，建议新增独立的告警动作接口，不要在当前列表接口里塞写操作
